# 9.15 - Routing Policy - Geoproximity

## Key Concepts

### 1. How Geoproximity Routing Works

- Routes traffic based on the **geographic location of users AND resources**
- Uses a **bias** value to shift traffic toward or away from a resource
- **Positive bias** (1 to 99) → attracts MORE traffic (expands the region's reach)
- **Negative bias** (-1 to -99) → attracts LESS traffic (shrinks the region's reach)
- **Bias = 0** → no shift, routes to the closest resource by location

### 2. The Bias Effect (Visualized)

```
Bias = 0 on both:
                    │
  ← us-west-1      │      us-east-1 →
                    │
  (even split based on proximity)


Bias = 0 on west, Bias = +50 on east:
              │
  ← us-west-1 │         us-east-1 →
              │
  (line shifts LEFT → east gets MORE traffic)
```

The bias essentially moves the "dividing line" — a positive bias on a region pulls that line toward the other regions, capturing more users.

### 3. Resource Types

| Resource | How Route 53 locates it |
|----------|------------------------|
| **AWS resource** | You specify the **AWS region** |
| **Non-AWS resource** (on-premises) | You specify **latitude & longitude** |

### 4. Requirements

- Requires **Route 53 Traffic Flow** (advanced feature) to use the bias
- Traffic Flow policy records cost **$50/month** each
- Traffic Flow policies can be **versioned** and applied to different hosted zones

---

## Exam Tips
- Geoproximity = **shift traffic between regions** using bias
- Positive bias → more traffic, Negative bias → less traffic
- Key exam signal: "need to shift/move traffic from one region to another" → **Geoproximity**
- Don't confuse with Geolocation (country/continent rules) — Geoproximity is about **proximity + bias adjustment**
- Non-AWS resources need latitude/longitude
- Requires Route 53 Traffic Flow
