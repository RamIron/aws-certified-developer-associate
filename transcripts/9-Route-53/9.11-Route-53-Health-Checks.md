# 9.11 - Route 53 Health Checks

## Key Concepts

### 1. Why Health Checks?
- Enable **automated DNS failover** — if a resource is unhealthy, Route 53 stops sending traffic to it
- Health checks are associated with DNS records
- Health checks have their own metrics visible in **CloudWatch**

### 2. Three Types of Health Checks

| Type | What it monitors |
|------|-----------------|
| **Endpoint** | A public resource (ALB, server, app) directly |
| **Calculated** | Combines multiple child health checks into one parent |
| **CloudWatch Alarm** | Monitors a CloudWatch Alarm (useful for **private resources**) |

### 3. Endpoint Health Checks

```
Route 53 Health Checkers (15+ worldwide)
    │         │         │
    ▼         ▼         ▼
  [Public Endpoint — ALB, EC2, etc.]
    │
    └── HTTP 2xx/3xx? → Healthy ✅
        Otherwise    → Unhealthy ❌
```

- **~15 global health checkers** send requests from around the world
- Healthy if **>18%** of checkers report healthy
- Interval: **30 seconds** (regular) or **10 seconds** (fast — higher cost)
- Protocols: **HTTP, HTTPS, TCP**
- Can check for specific **text** in the first 5,120 bytes of the response
- **Important:** your security group must allow incoming traffic from Route 53 health checker IPs

### 4. Calculated Health Checks

```
[Child HC 1] ──┐
[Child HC 2] ──┼──► [Parent Health Check] ──► Combined result
[Child HC 3] ──┘
                    Conditions: OR, AND, NOT
```

- Combine up to **256 child health checks**
- You specify how many must pass for the parent to pass
- Use case: perform maintenance on one instance without triggering a full failover

### 5. Private Resource Health Checks

Route 53 health checkers are **public** — they can't access private VPC resources. Workaround:

```
[EC2 in private subnet] ──► CloudWatch Metric ──► CloudWatch Alarm ──► Health Check
                                                    (alarm state = unhealthy)
```

1. Monitor the private resource with a **CloudWatch Metric**
2. Create a **CloudWatch Alarm** on that metric
3. Create a health check that monitors the **CloudWatch Alarm**
4. Alarm triggers → health check = unhealthy

---

## Exam Tips
- Health checks enable **automated DNS failover**
- 3 types: Endpoint (public), Calculated (combine children), CloudWatch Alarm (private resources)
- Endpoint: ~15 checkers, >18% must say healthy, interval 30s or 10s (fast)
- Must allow Route 53 health checker IPs in your **security group**
- **Private resources** → use CloudWatch Metric → Alarm → Health Check
- Calculated: up to 256 children, combine with OR/AND/NOT
- Can check text in first 5,120 bytes of response
