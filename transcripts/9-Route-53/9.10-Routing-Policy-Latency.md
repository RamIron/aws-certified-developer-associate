# 9.10 - Routing Policy - Latency

## Key Concepts

### 1. How Latency-Based Routing Works
- Redirects users to the resource with the **lowest latency** (closest region)
- Latency is measured based on how fast the user can connect to an **AWS region**
- You must specify the **region** for each record (Route 53 doesn't auto-detect it from the IP)
- Supports **health checks**

```
                                    ┌──► ALB (us-east-1)
User (New York) ──► Route 53 ──────┤    ← lowest latency, picked!
                                    ├──► ALB (eu-central-1)
                                    └──► ALB (ap-southeast-1)

                                    ┌──► ALB (us-east-1)
User (Tokyo) ──► Route 53 ─────────┤
                                    ├──► ALB (eu-central-1)
                                    └──► ALB (ap-southeast-1)
                                         ← lowest latency, picked!
```

### 2. Key Details
- Latency is based on the user's **network path** to the AWS region, not geographic distance
- A user in Germany could be routed to the US if latency is lower there (unlikely but possible)
- Each record needs a **region** specified — Route 53 uses this to calculate latency
- Very common routing policy for global applications

---

## Exam Tips
- Latency-based = route to the **lowest latency** AWS region for the user
- You specify the region per record — Route 53 evaluates latency to that region
- Latency ≠ geographic distance (it's about network speed, not proximity)
- Supports health checks
- Use when **latency is the priority** for your users
