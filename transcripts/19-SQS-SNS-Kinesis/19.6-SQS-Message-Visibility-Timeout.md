# 19.6 - SQS Message Visibility Timeout

## Key Concepts

### 1. What Is Visibility Timeout?
- When a consumer **polls** a message from SQS, the message becomes **invisible** to other consumers
- This "hidden" period is the **visibility timeout**
- Default: **30 seconds** | Range: **0 seconds to 12 hours**

```
SQS Queue: [Msg A] [Msg B] [Msg C]
Consumer polls Msg A
→ Msg A disappears from queue (invisible to others)
→ Consumer has 30s (default) to process and call DeleteMessage
```

### 2. Two Risk Scenarios

| Scenario | Problem | Fix |
|----------|---------|-----|
| Timeout **too short** | Consumer hasn't finished → timeout expires → message becomes visible again → **another consumer picks it up → processed twice** | Increase the timeout |
| Timeout **too high** | Consumer crashes mid-processing → message only reappears after (e.g.) 12 hours → **long delay** before another consumer can retry | Lower the timeout |

### 3. ChangeMessageVisibility API
- Consumer can **extend** the visibility timeout mid-flight before it expires
- Use when the consumer knows it needs more processing time
- Prevents the message from becoming visible while it's still being processed

```
Consumer polls Msg A (30s timeout starts)
→ At 20s, consumer calls ChangeMessageVisibility (extends to 60s more)
→ Consumer finishes processing at 45s → calls DeleteMessage ✅
```

## Exam Tips
- Default visibility timeout = **30 seconds**
- Range: **0 to 12 hours**
- Message processed twice = visibility timeout **too short**
- Long delay after consumer crash = visibility timeout **too high**
- **`ChangeMessageVisibility`** = API to extend timeout while message is in flight
- Best practice: set a reasonable timeout + use `ChangeMessageVisibility` for unpredictable processing times
