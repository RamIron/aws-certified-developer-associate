# 19.2 - Amazon SQS - Standard Queues Overview

## Key Concepts

### 1. Core Model — Producer → Queue → Consumer
- **Producer**: sends messages into the queue using the SDK (`SendMessage` API)
- **Queue**: holds messages until a consumer reads and deletes them
- **Consumer**: polls the queue, processes messages, then deletes them (`DeleteMessage` API)
- Multiple producers and multiple consumers are supported
- Consumers can be EC2 instances, on-premises servers, or Lambda functions

### 2. Standard Queue Characteristics
| Property | Value |
|----------|-------|
| Throughput | Unlimited (messages/sec) |
| Queue size | Unlimited (number of messages) |
| Message retention | Default **4 days**, max **14 days** |
| Max message size | **1,024 KB** |
| Latency | < 10 ms on publish and receive |
| Delivery | **At-least-once** (duplicates possible) |
| Ordering | **Best-effort** (not guaranteed) |

### 3. Consumers — Poll, Process, Delete
- Consumer polls the queue and receives **up to 10 messages at a time**
- Consumer processes the message (e.g. insert into RDS)
- Consumer calls `DeleteMessage` to remove it from the queue
- If a message is not deleted, other consumers can receive it → at-least-once delivery

### 4. Scaling with ASG + CloudWatch
- Consumers run on EC2 instances inside an **Auto Scaling Group**
- CloudWatch metric: **`ApproximateNumberOfMessages`** (queue length)
- Set a CloudWatch Alarm → when queue length exceeds threshold → scale out the ASG
- More messages in queue → more consumers → higher throughput → queue drains

```
SQS Queue ──► CloudWatch metric (ApproximateNumberOfMessages)
                      ↓
              CloudWatch Alarm (queue length > X)
                      ↓
              ASG scales out → more EC2 consumers
```

### 5. Decoupled Architecture Use Case
- Split application into **front-end tier** (receives requests) and **back-end tier** (processes them)
- Front-end sends a message to SQS instead of doing the heavy work itself
- Back-end ASG reads from the queue and does the actual processing (e.g. video encoding)
- Each tier scales independently, uses optimal instance types for its workload

```
Users → Front-end ASG → SQS Queue → Back-end ASG (GPU instances) → S3
```

### 6. SQS Security
| Type | Details |
|------|---------|
| **In-flight encryption** | HTTPS API (always on) |
| **At-rest encryption** | KMS keys (optional) |
| **Client-side encryption** | Supported but client manages it |
| **Access control** | IAM policies on the SQS API |
| **SQS Access Policies** | Like S3 bucket policies — for cross-account access and allowing other AWS services (SNS, S3) to write to the queue |

## Exam Tips
- SQS = **decouple applications** — the go-to answer whenever you see "decouple" in a question
- Standard queue: **at-least-once delivery** (duplicates possible) + **best-effort ordering**
- Message retention: default **4 days**, max **14 days**
- Max message size: **1,024 KB**
- Consumers receive **up to 10 messages at a time**
- Scale consumers with ASG using **`ApproximateNumberOfMessages`** CloudWatch metric
- SQS Access Policies → cross-account access or letting SNS/S3 write to a queue
