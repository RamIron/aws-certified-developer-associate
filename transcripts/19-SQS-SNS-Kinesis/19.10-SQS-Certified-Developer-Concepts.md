# 19.10 - SQS Certified Developer Concepts

## Key Concepts

### 1. Long Polling
- When a consumer polls an **empty queue**, it can **wait** for messages to arrive instead of returning immediately
- Reduces API calls → less cost, less CPU cycles
- Decreases latency — message is delivered to consumer as soon as it arrives in the queue
- Wait time: **1 to 20 seconds** (20 seconds recommended)
- Short polling = 0 seconds (default) = returns immediately even if empty

```
Short Polling (default):
Consumer polls → Queue empty → Empty response immediately
Consumer polls → Queue empty → Empty response immediately
Consumer polls → Queue empty → Empty response immediately
(wasted API calls!)

Long Polling (20s):
Consumer polls → Queue empty → Waits up to 20s...
  ...message arrives at 8s → Response with message ✅
(1 API call instead of many)
```

**Two ways to enable:**
| Method | Scope |
|--------|-------|
| Queue-level: set **Receive Message Wait Time** > 0 | All consumers on this queue |
| API-level: use `ReceiveMessageWaitTimeSeconds` parameter | Per API call |

### 2. SQS Extended Client (Large Messages)
- Max message size = **1,024 KB** — so how to send large messages (e.g., 1 GB video)?
- Use the **SQS Extended Client** (Java library, but the pattern works in any language)
- Pattern: upload large payload to **S3**, send a **small metadata message** (with S3 pointer) to SQS

```
Producer                         S3 Bucket
   │                                │
   ├── Upload large file (1GB) ────►│ [video.mp4]
   │                                │
   ├── Send small metadata msg ──►  SQS Queue: [pointer to S3]
   │                                │
Consumer                            │
   │                                │
   ├── Poll SQS → gets metadata ◄──┤
   ├── Read large file from S3  ◄───│ [video.mp4]
   └── Process ✅
```

### 3. Must-Know API Calls

| API | Purpose |
|-----|---------|
| `CreateQueue` | Create a queue (use `MessageRetentionPeriod` arg) |
| `DeleteQueue` | Delete queue + all its messages |
| `PurgeQueue` | Delete all messages (keep the queue) |
| `SendMessage` | Send a message (use `DelaySeconds` for delay) |
| `ReceiveMessage` | Poll messages (default: 1, max: **10** via `MaxNumberOfMessages`) |
| `DeleteMessage` | Delete a message after processing |
| `ChangeMessageVisibility` | Extend visibility timeout mid-processing |

**Batch variants exist for:** `SendMessage`, `DeleteMessage`, `ChangeMessageVisibility` → reduces API calls and costs

**Why batch `ChangeMessageVisibility`?** If a consumer polled 10 messages at once (`MaxNumberOfMessages = 10`) and needs more time for all of them, one batch call extends all 10 timeouts instead of making 10 separate API calls.

## Exam Tips
- "Too many API calls / high cost polling" → **Long polling** (set `ReceiveMessageWaitTimeSeconds` or queue-level wait time)
- Long polling range: **1–20 seconds** (20 recommended)
- Large messages > 1,024 KB → **SQS Extended Client** + **S3**
- `MaxNumberOfMessages` default = **1**, max = **10**
- Batch APIs available for Send, Delete, ChangeVisibility → **cost optimization**
