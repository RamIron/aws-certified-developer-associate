# 19.8 - SQS Dead Letter Queues - Hands On

## Exam-Relevant Observations

### 1. DLQ Setup Recap
- Created a DLQ with **14 days retention** (max, as recommended)
- Source queue configured with **MaximumReceives = 3** → after 3 failed reads, message moves to DLQ
- Visibility timeout set to **5 seconds** (low for demo purposes)

### 2. Simulating Consumer Failure
- Stephane set a low visibility timeout (5s) and never called `DeleteMessage`
- The message was polled 3 times without being deleted → hit MaximumReceives → moved to DLQ automatically
- This simulates what happens in real life when a consumer crashes or throws an exception before reaching `DeleteMessage`

### 3. Redrive to Source (Console)
- In the DLQ console: **"Start DLQ redrive"** button (top right)
- Option: **Redrive to source queue**
- Velocity control: **System optimized** (default)
- After redrive: message reappears in the source queue for reprocessing
