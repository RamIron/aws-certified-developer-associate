# 19.13 - Amazon SNS

## Key Concepts

### 1. Pub/Sub Pattern
- **Problem:** Direct integration (service A sends to B, C, D individually) is cumbersome — adding a new receiver means writing a new integration
- **Solution:** Pub/Sub — producer publishes to an **SNS topic**, all **subscribers** receive the message automatically

```
Direct Integration (bad):
Buying Service ──► Email Service
               ──► Fraud Service
               ──► Shipping Service
               ──► SQS Queue
(add a new one? write new code)

Pub/Sub (good):
Buying Service ──► [SNS Topic] ──► Email Service
                               ──► Fraud Service
                               ──► Shipping Service
                               ──► SQS Queue
(add a new one? just subscribe)
```

### 2. Limits
- Up to **12,500,000+ subscriptions** per topic
- Up to **100,000 topics** per account
- (Limits can change and are not tested on the exam)

### 3. Subscribers (Who Can Receive — from AWS Console)
- **Amazon Kinesis Data Firehose**
- **Amazon SQS**
- **AWS Lambda**
- **Email** (plain text)
- **Email-JSON** (structured JSON format)
- **HTTP** / **HTTPS** endpoints
- **SMS**

> Note: Stephane mentioned Mobile push notifications in the theory, but the current console shows the list above. Email-JSON is a separate protocol from plain Email.

### 4. Publishers (Who Can Send to SNS)
- Many AWS services publish to SNS natively: CloudWatch Alarms, ASG notifications, CloudFormation state changes, S3 events, Budgets, DMS, Lambda, DynamoDB, RDS events, etc.

### 5. How to Publish
| Method | Use Case |
|--------|----------|
| **Topic Publish** (SDK) | Create topic → create subscriptions → publish to topic → all subscribers receive |
| **Direct Publish** (Mobile SDK) | Create platform application → create platform endpoint → publish to endpoint (for GCM, APNS, ADM) |

### 6. Security (Same as SQS)
- **In-flight encryption:** HTTPS (enabled by default)
- **At-rest encryption:** KMS keys
- **Client-side encryption:** Producer responsibility
- **Access control:** IAM policies for API access
- **SNS Access Policies:** Similar to S3 bucket policies — cross-account access and allowing other services (e.g., S3) to write to SNS topics

## Exam Tips
- SNS = **Pub/Sub** pattern (one message, many receivers)
- SQS = **Queue** pattern (one message, one consumer)
- SNS security model = **identical to SQS** (encryption + IAM + resource policies)
- AWS services can publish directly to SNS (CloudWatch Alarms, S3 events, etc.)
