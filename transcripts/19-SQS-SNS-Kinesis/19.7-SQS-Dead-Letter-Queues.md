# 19.7 - SQS Dead Letter Queues (DLQ)

## Key Concepts

### 1. The Problem: Failure Loop
- Consumer fails to process a message → message goes back into the queue
- If the message is consistently unprocessable (e.g., malformed), this creates an **infinite loop**
- Solution: set a **MaximumReceives threshold** — if exceeded, move the message to a **Dead Letter Queue (DLQ)**

```
Source Queue: [Bad Msg] ←────────────────┐
                  ↓                       │
              Consumer fails              │
                  ↓                       │
          Message goes back         (loop repeats)
          to queue (visible)             │
                  └────────────────────►─┘

          After MaximumReceives threshold exceeded:
                  ↓
          Dead Letter Queue: [Bad Msg] ← stored for debugging
```

### 2. Dead Letter Queue Rules
- DLQ of a **FIFO queue** must also be a **FIFO queue**
- DLQ of a **Standard queue** must also be a **Standard queue**
- ⚠️ If a message **expires** in the source queue before reaching MaximumReceives, it's **lost forever** — it never reaches the DLQ
- Set **retention period to 14 days** in the DLQ so you have maximum time to debug messages that made it there

### 3. Redrive to Source
- After debugging and fixing your consumer, you can **redrive** messages from the DLQ back to the source queue
- Consumer reprocesses the message without knowing it ever went to a DLQ
- Useful when the issue was the consumer code, not the message itself

## Exam Tips
- **MaximumReceives** = threshold that triggers DLQ routing
- DLQ is just a regular SQS queue — must match type (FIFO→FIFO, Standard→Standard)
- DLQ retention = **14 days** recommended (max retention)
- **Redrive to source** = move messages from DLQ back to source queue for reprocessing
- Purpose of DLQ: **debugging** — inspect messages that couldn't be processed
