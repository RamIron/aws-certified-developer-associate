# 19.16 - Amazon Kinesis Data Streams

## Key Concepts

### 1. What Is Kinesis Data Streams?
- Service to **collect and store streaming data in real-time**
- Exam keyword: **real-time**
- Use cases: click streams, IoT device data, metrics/logs

```
Producers                    Kinesis Data Streams          Consumers
─────────                    ────────────────────          ─────────
Applications (SDK)     ──►                           ──►  Applications (SDK)
Kinesis Agent          ──►   [Shard 1] [Shard 2] ... ──►  Lambda
                                                     ──►  Amazon Data Firehose
                                                     ──►  Managed Service for Apache Flink
```

### 2. Core Properties
- Retention: **1 day (default) up to 365 days**
- Data is **immutable** — once sent, cannot be deleted (must wait for expiration)
- Max data size: **10 MB** per record (but typical use = many small records)
- Data is **ordered within the same partition ID** (similar to SQS FIFO Message Group ID)
- Ability to **reprocess/replay data** (unlike SQS where message is deleted after processing)

### 3. Shards (Provisioned Mode)
- A **shard** = unit of **throughput** (how much data flows through), NOT processing power
- More shards = wider highway, more data in/out per second
- If your consumer is slow → scale the consumer, not the shards
- If data is coming in faster than the stream can handle → add more shards
- Per shard throughput:

| Direction | Throughput |
|-----------|-----------|
| **In** (write) | 1 MB/s OR 1,000 records/s |
| **Out** (read) | 2 MB/s |

```
Need 10 MB/s inbound? → Provision 10 shards
Need 10,000 records/s? → Provision 10 shards
```

- Scale **manually** — but can only **double** shards per operation (1 → 2 → 4 → 8...)
- Pay **per shard provisioned per hour**

### 4. On-Demand Mode
- No provisioning or capacity management
- Default: **4 MB/s in** (or 4,000 records/s), **8 MB/s out** (same 2:1 ratio as provisioned)
- Auto-scales based on observed throughput from the **past 30 days**
- Pay **per stream per hour** + data in/out volume

### 5. Optimized Libraries
| Library | Purpose |
|---------|---------|
| **KPL** (Kinesis Producer Library) | High-throughput producer applications |
| **KCL** (Kinesis Client Library) | Optimized consumer applications |

### 6. Two Consumption Modes (Standard vs Enhanced Fan-Out)

| Feature | Standard | Enhanced Fan-Out |
|---------|----------|-----------------|
| **Model** | Consumer **pulls** data | Kinesis **pushes** data to consumer |
| **Throughput** | 2 MB/s per shard **shared** across all consumers | 2 MB/s per shard **per consumer** (dedicated) |
| **Use when** | Few consumers | Many consumers needing high throughput |
| **Cost** | Lower | Higher |

```
Standard (3 consumers, 1 shard):
  2 MB/s ÷ 3 consumers = ~0.67 MB/s each

Enhanced Fan-Out (3 consumers, 1 shard):
  2 MB/s × 3 consumers = 6 MB/s total
```

### 7. Security
- At-rest: **KMS encryption**
- In-flight: **HTTPS encryption**

## Exam Tips
- **Real-time** in the question → think Kinesis Data Streams
- Data is **immutable** and can be **replayed** (unlike SQS)
- Ordering via **partition ID** (similar concept to SQS FIFO Message Group ID)
- Provisioned mode: **1 MB/s in, 2 MB/s out per shard** — scale manually
- On-demand: **4 MB/s in** default, auto-scales on 30-day history
- KPL = producer, KCL = consumer
- **Standard** consumption = pull, 2 MB/s shared per shard
- **Enhanced Fan-Out** = push, 2 MB/s dedicated per consumer per shard
