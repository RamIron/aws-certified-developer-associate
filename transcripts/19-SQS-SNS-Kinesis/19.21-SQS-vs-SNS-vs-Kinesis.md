# 19.21 - SQS vs SNS vs Kinesis

## The Big Comparison Table

| Feature | SQS | SNS | Kinesis Data Streams |
|---------|-----|-----|---------------------|
| **Model** | Consumer **pulls** data | **Push** to many subscribers | **Pull** (standard) or **Push** (enhanced fan-out) |
| **After processing** | Consumer **deletes** message | Not persisted — if not delivered, it's lost | Data **persists** (1–365 days), can replay |
| **Consumers** | As many as you want, all sharing work | Up to 12.5M subscribers per topic | Standard: 2 MB/s per shard shared; Enhanced fan-out: 2 MB/s per shard **per consumer** |
| **Throughput** | No provisioning needed, auto-scales | No provisioning needed, auto-scales | Provisioned (manual shards) or On-demand |
| **Ordering** | Only with **FIFO queues** | Only with **FIFO topics** | Ordered at **shard level** (partition key) |
| **Message delay** | Yes — per-message `DelaySeconds` | No | No |
| **Replay** | ❌ No (deleted after processing) | ❌ No (not persisted) | ✅ Yes |
| **Use case** | Decouple services, work queues | Pub/sub notifications, fan out | Real-time big data, analytics, ETL |
| **Combine with** | SNS (fan out pattern) | SQS (fan out), SQS FIFO | Data Firehose, Apache Flink, Lambda |

## Quick Decision Guide

```
"Decouple microservices, process tasks"     → SQS
"Send one event to many receivers"          → SNS
"Fan out to multiple queues"                → SNS + SQS
"Real-time streaming, analytics, replay"    → Kinesis Data Streams
"Load streaming data into S3/Redshift"      → Data Firehose
"Real-time stream transformation"           → Apache Flink
```

## Exam Keyword Triggers
- **"pull", "decouple", "work queue"** → SQS
- **"push", "pub/sub", "fan out", "many subscribers"** → SNS
- **"real-time", "big data", "replay", "analytics", "ETL"** → Kinesis
- **"near real-time", "load into S3/Redshift"** → Data Firehose
- **"not persisted", "lost if not delivered"** → SNS (this is a trap!)
