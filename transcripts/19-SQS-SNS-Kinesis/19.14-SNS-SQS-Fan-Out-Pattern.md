# 19.14 - Amazon SNS and SQS - Fan Out Pattern

## Key Concepts

### 1. The Fan Out Pattern
- **Problem:** Sending the same message to multiple SQS queues individually is fragile — app crashes, delivery failures, hard to add new queues
- **Solution:** Push once to an SNS topic → subscribe multiple SQS queues

```
Without Fan Out (fragile):
Buying Service ──► SQS Queue 1 (Fraud)
               ──► SQS Queue 2 (Shipping)
               ──► SQS Queue 3 (???) ← crashes mid-send, lost!

With Fan Out (robust):
Buying Service ──► [SNS Topic] ──► SQS Queue 1 (Fraud)
                               ──► SQS Queue 2 (Shipping)
                               ──► SQS Queue 3 (new, just subscribe!)
```

- Fully decoupled, no data loss
- SQS provides: data persistence, delayed processing, retries
- Cross-region delivery supported (SNS in region A → SQS in region B)
- Requirement: SQS queue access policy must allow the SNS topic to write

### 2. S3 Events → Fan Out
- S3 event rule limitation: for a given **event type + prefix** combination, you can only have **one rule**
- To send the same S3 event to multiple destinations → use fan out: S3 → SNS → multiple SQS queues (+ Lambda, Email, etc.)

```
S3 (object created, /images/) ──► [SNS Topic] ──► SQS Queue 1
                                               ──► SQS Queue 2
                                               ──► Lambda
                                               ──► Email
```

### 3. SNS → Kinesis Data Firehose (KDF) → S3
- SNS has a direct integration with Kinesis Data Firehose
- Allows persisting SNS messages to S3 (or any KDF-supported destination)

```
Buying Service ──► [SNS Topic] ──► KDF ──► S3 Bucket
```

### 4. SNS FIFO Topics
- SNS also supports **FIFO** — ordering by Message Group ID + deduplication
- Subscribers: **only SQS queues** (FIFO and Standard) — no Lambda, Email, HTTP, etc.
- Same throughput limits as SQS FIFO (300/3,000 msg/s)
- Use case: **fan out + ordering + deduplication**

```
Buying Service ──► [SNS FIFO Topic] ──► SQS FIFO Queue (Fraud)
                                    ──► SQS FIFO Queue (Shipping)
```

### 5. SNS Message Filtering
- JSON filter policy applied to an SNS **subscription** (not the topic itself)
- Only messages matching the filter are delivered to that subscriber
- **No filter policy = receives ALL messages** (default)

```json
// Filter: only "placed" orders
{"State": ["placed"]}
```

```
SNS Topic (all transactions)
  ├── SQS Queue (filter: State=placed)    → only placed orders
  ├── SQS Queue (filter: State=cancelled) → only cancelled orders
  ├── Email    (filter: State=cancelled)  → email on cancellation
  ├── SQS Queue (filter: State=declined)  → only declined orders
  └── SQS Queue (no filter)              → ALL messages
```

## Exam Tips
- **Fan out** = SNS + SQS for sending one message to many queues reliably
- S3 event rule **one rule per event type + prefix** → fan out to send to many
- SNS → **KDF → S3** for persisting messages
- SNS FIFO subscribers: **only SQS queues** (Standard and FIFO) — no Lambda, Email, HTTP, etc.
- **Message filtering** = JSON policy on the subscription, not the topic
- No filter = receives everything
