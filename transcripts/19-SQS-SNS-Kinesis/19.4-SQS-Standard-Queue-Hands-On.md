# 19.4 - SQS Standard Queue - Hands On

## Key Takeaways

### Encryption Options in the Console
Two server-side encryption options when creating a queue:

| Option | Details |
|--------|---------|
| **SSE-SQS** | Uses an Amazon SQS-managed key — similar to SSE-S3 in S3. Default option. |
| **SSE-KMS** | Uses a KMS Customer Master Key (CMK). Allows choosing the key and setting a **Data Key Reuse Period** (e.g. 5 min) to limit the number of API calls made to KMS. |

### Purge Queue
- Deletes **all messages** in the queue at once
- Useful during development to reset state
- Should **not** be used in production

### Visibility Timeout — Confirmed Default
- Demonstrated in the console: a message received but not deleted becomes visible again after **30 seconds**
- The receive count on the message increments each time it is re-received

### Additional Scaling Metric
- **Approximate age of oldest message** — another CloudWatch metric available on any SQS queue
- Can be used (alongside `ApproximateNumberOfMessages`) to trigger ASG scaling
- Useful when you care about message latency, not just backlog size

## Exam Tips
- **SSE-SQS** = SQS-managed encryption key (default, similar to SSE-S3)
- **SSE-KMS** = customer-managed key, with configurable data key reuse period to reduce KMS API calls
- **Purge** = wipe all messages from the queue — useful in dev, dangerous in prod
- Default visibility timeout = **30 seconds**
