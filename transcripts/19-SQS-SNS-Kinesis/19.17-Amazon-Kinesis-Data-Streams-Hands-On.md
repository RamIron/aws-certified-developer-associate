# 19.17 - Amazon Kinesis Data Streams - Hands On

## Exam-Relevant Observations

### 1. On-Demand Mode Console Details
- Max write: **200 MB/s** (200,000 records/s)
- Max read: **400 MB/s** per consumer (with enhanced Fan-Out)
- Pay-per-throughput pricing, **no free tier**

### 2. Producer Options (shown in console)
| Option | Level |
|--------|-------|
| **Kinesis Agent** | Install on servers to stream logs/data |
| **AWS SDK** | Low-level, manual control |
| **KPL** (Kinesis Producer Library) | High-level, better API |

### 3. Consumer Options (shown in console)
- Kinesis Data Analytics (Apache Flink)
- Kinesis Data Firehose
- **KCL** (Kinesis Client Library)
- **Lambda** (not shown in console but supported)

### 4. Key API — `put-record`
- API to send a single record to a stream
- Requires: stream name, **partition key**, data
- Text data must use `--cli-binary-format raw-in-base64-out`
- Data with the **same partition key** → same shard (ordering guarantee)

### 5. Consuming at Low Level (SDK/CLI)
- **Step 1:** `describe-stream` → get shard IDs
- **Step 2:** `get-shard-iterator` → get a pointer to start reading
  - `TRIM_HORIZON` = read from the **beginning** of the stream
  - `LATEST` = read only **new records** from now on
- **Step 3:** `get-records --shard-iterator <iterator>` → get batch of records
- **Step 4:** Use `NextShardIterator` from response for next read
- Data comes back **base64-encoded** — must decode
- This is the **shared consumption mode** (not enhanced Fan-Out)

### 6. Enhanced Fan-Out
- Mentioned but not demoed
- Should use **KCL** for a proper consumer implementation
- Provides dedicated **2 MB/s per consumer per shard** (instead of sharing 2 MB/s across all consumers)

### 7. Scaling (Shard Splitting)
- You can only **double** shards in one operation (e.g., 1 → 2, not 1 → 5)
