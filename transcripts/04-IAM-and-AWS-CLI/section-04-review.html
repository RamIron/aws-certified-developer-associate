IAM Basics

🌐

Global Service

IAM is not region-specific. Users, groups, roles, and policies you create are available across all AWS regions.

👑

Root Account

Created automatically when you open an AWS account. Has unlimited power. Only use for initial setup — never share, never use daily.

👤

Users

Represent individual people or services. One user = one person. Users have no permissions by default.

👥

Groups

Organize users together. Policies attached to a group are inherited by all users in it. Groups can only contain users — not other groups.

IAM Hierarchy — Users, Groups & Membership

Alice is a member of both Admins and Developers groups — a user can belong to zero or multiple groups simultaneously.
⚠ Groups can only contain users — never other groups. This is a common exam trap.

IAM Policies

Policies are JSON documents that define what actions are allowed or denied on which AWS resources. They are the core of IAM permissions.

Policy JSON Structure

{ "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", // Allow or Deny "Principal": { "AWS": "arn:aws:iam::123456789012:root" }, // Who "Action": "s3:GetObject", // API call(s) "Resource": "arn:aws:s3:::my-bucket/*", // What resource "Condition": { ... } // Optional: when to apply } ] }

Policy Fields Reference

Field Values Description
Effect Allow / Deny Whether the statement grants or blocks access
Principal Account / User / Role ARN Who the policy applies to (only in resource-based policies)
Action e.g. s3:GetObject, ec2:* The specific API calls being permitted or denied
Resource ARN or * (wildcard) The AWS resource(s) the action targets
Condition Key-value expressions Optional — restricts when the policy applies (IP, time, MFA, etc.)

Policy Inheritance — Weak Point

🔴 Weak point area: Understand how policies stack.
  • Group policy — attached to a group; all users in that group inherit it automatically.
  • Inline policy — attached directly to a single user; applies only to that user.
  • Both apply simultaneously — a user's effective permissions = all group policies + all inline policies combined.
  • Deny overrides Allow — an explicit Deny anywhere will block the action, even if another policy Allows it.
Best practice: Assign permissions via groups, not individual users. Inline policies are exceptions — use them sparingly. Apply the least privilege principle: grant only what is needed.

Password Policy & MFA

AWS lets you enforce a password policy for IAM users: minimum length, required character types, expiration rules, and prevent password reuse. MFA adds a second layer of security.

MFA Device Types

📱

Virtual MFA Device

Software

Authenticator apps like Google Authenticator or Authy. One device can hold multiple MFA tokens for different accounts.

🔑

U2F Security Key

Physical

Hardware USB key (e.g. YubiKey by Yubico). One physical key can support multiple IAM users across multiple accounts.

🔒

Hardware Key Fob

Physical

Dedicated hardware token by Gemalto (third-party provider). Generates a time-based one-time password.

🏛

GovCloud Key Fob

GovCloud Only

Hardware token by SurePassID. Available exclusively for AWS GovCloud (US) regions.

Always enable MFA for the root account. This is non-negotiable security hygiene — root account compromise gives unlimited access to everything in your AWS account.

Access Key Concepts

Access Key ID ≈ Username — identifies who you are
Secret Access Key ≈ Password — proves you own the key
🚨 Never share access keys. Never commit them to git. Each user manages their own keys — you can only view the Secret Access Key once at creation time. If lost, delete and recreate.
The AWS CLI is built on top of the Python SDK (Boto3). When you run aws s3 ls, it ultimately calls the same API as Boto3's s3.list_buckets().

IAM Roles

Roles are like users, but designed for AWS services — not people. When a service assumes a role, AWS provides temporary credentials automatically. No access keys needed (or allowed).

🖥

EC2 Instance Role

Allows EC2 instances to call AWS APIs (e.g., read from S3, write to DynamoDB) without storing credentials on the instance.

λ

Lambda Function Role

Lambda automatically assumes an execution role on invocation. Defines what AWS services the Lambda function is allowed to call.

🏗

CloudFormation Role

Grants CloudFormation permission to create, update, or delete AWS resources on your behalf when deploying stacks.

🔄

Cross-Account Role

Allows principals in one AWS account to assume a role in another account, providing controlled cross-account access without sharing credentials.

🚨 NEVER put access keys on an EC2 instance. If the instance is compromised, those keys are compromised. Use an IAM Role instead — EC2 gets temporary, automatically rotating credentials with no manual key management.

IAM Security Tools

AWS provides two built-in tools to audit IAM permissions and enforce the least privilege principle.

Credentials Report

Account-level
  • Lists all IAM users in the account
  • Shows credential status for each user
  • Includes password age, last used, MFA enabled
  • Shows access key age and last rotation
  • Download as CSV from IAM console
  • Use to identify stale or risky credentials

Access Advisor

User-level
  • Shows service-level permission access history
  • Displays when each service was last accessed
  • Available per user, group, role, or policy
  • Identify services never accessed → remove them
  • Primary tool for enforcing least privilege
  • Data retained for up to 400 days
Least privilege workflow: Use Access Advisor to see what services a user actually accessed → remove permissions for everything they never used.

Exam Tips

"IAM is" → Global service (no region selection)
"Groups can contain" → Users only — NOT other groups
"A user can be in" → Zero or multiple groups simultaneously
"Roles are for" → AWS services (not people) — temporary credentials
"NEVER put on EC2" → Access keys — use IAM Roles instead
"Connection timeout to EC2" → Security Group is blocking the port
"Connection refused on EC2" → App problem — traffic reached the instance
"Credentials Report scope" → Account-level (all users + credential status)
"Access Advisor scope" → User-level (one user's service access history)
"Least privilege tool" → Access Advisor — remove unused service permissions
