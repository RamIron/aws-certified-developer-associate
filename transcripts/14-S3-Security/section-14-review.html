01 — S3 Encryption Methods
4 Server-Side & Client-Side Encryption Methods

S3 supports four distinct encryption approaches. Each has different key ownership, auditability, and performance characteristics. The exam tests which header, which key owner, and when to use each.

Default Encryption (2023+): All new S3 buckets and objects use SSE-S3 by default. You can change this to SSE-KMS at bucket level. Bucket policies are evaluated BEFORE default encryption settings are applied. To force HTTPS, use a bucket policy with the aws:SecureTransport condition key.
01b — Encryption Quick Reference
Side-by-Side Comparison
Method Key Owner Header Value HTTPS Required? Console Support? Audit Trail?
SSE-S3 AWS (S3-managed) AES256 Optional Yes No
SSE-KMS You (via KMS) aws:kms Optional Yes CloudTrail
SSE-C You (per-request) Your key in headers MANDATORY No (CLI/SDK) No
Client-Side You (local) N/A (pre-encrypted) Optional Yes (upload blob) No
KMS Throttling Trap: SSE-KMS calls KMS on every upload AND download. KMS quota is 5,000–30,000 requests/second per region. High-throughput S3 buckets can hit this limit. Consider S3 Bucket Keys (reduces KMS API calls by ~99%) to mitigate throttling.
02 — S3 CORS
Cross-Origin Resource Sharing

CORS is a web browser security mechanism — it is enforced by the browser, not the server. An "origin" is defined as: scheme + host + port (e.g. https://example.com:443).

How CORS works with S3
  • Browser on Origin A makes request to S3 bucket (Origin B)
  • Browser sends Origin header to S3
  • S3 checks its CORS configuration
  • S3 responds with Access-Control-Allow-Origin
  • If header matches, browser allows the response
Key Rules
  • Configure CORS on the TARGET bucket (the one being called)
  • Allow a specific origin or use * for all origins
  • NOT a server-side security mechanism
  • Pre-flight OPTIONS request sent first for non-simple requests
  • Common exam scenario: static website on S3 calling another S3 bucket

03 — MFA Delete
S3 MFA Delete — Extra Deletion Protection

MFA Delete adds an extra layer of protection against accidental or malicious permanent deletion of versioned objects.

Prerequisites & Management
  • Versioning MUST be enabled on the bucket
  • Only the ROOT ACCOUNT can enable or disable MFA Delete
  • Only manageable via CLI — not the AWS Console
  • Requires a hardware or virtual MFA device
When is MFA Required?
  • MFA Required — Permanently delete an object version
  • MFA Required — Suspend versioning on the bucket
  • MFA NOT Required — Enable versioning
  • MFA NOT Required — List object versions
  • MFA NOT Required — Add a delete marker (soft delete)
Exam Trap: MFA Delete protects against permanent deletion of specific versions. A regular delete (without version ID) only creates a delete marker — no MFA needed for that. The root account restriction means IAM admins cannot manage MFA Delete without root credentials.
04 — S3 Access Logs
S3 Server Access Logging

S3 Access Logs capture all requests made to a bucket — successful and failed, from any account. Essential for auditing, security analysis, and compliance.

What Gets Logged
  • All requests (GET, PUT, DELETE, HEAD)
  • Authorized AND denied requests
  • Requests from any AWS account
  • Requester, bucket, key, action, response status
  • Error codes when applicable
Configuration Rules
  • Stored in a different S3 bucket (same region)
  • Analyze with Amazon Athena (SQL queries on log files)
  • Log format is AWS log format (parseable)
  • No realtime — logs delivered within a few hours
Infinite Loop Warning — Critical Exam Trap: NEVER configure the logging target bucket to be the SAME as the monitored bucket. If you do, S3 writes a log entry for every log write, triggering another log write, infinitely. Result: exponential storage growth and a massive, unexpected AWS bill.

05 — Pre-signed URLs
S3 Pre-signed URLs — Temporary Access

A pre-signed URL embeds temporary credentials into the URL itself. The URL recipient inherits the permissions of the IAM user or role that generated it.

How It Works
  • Generator signs the URL with their AWS credentials + expiry
  • Recipient uses URL directly — no AWS account needed
  • Inherits generator's S3 permissions
  • Works for GET (download) and PUT (upload)
  • URL expires after the configured duration
Common Use Cases
  • Serve private/premium S3 content to logged-in users
  • Allow one-time file download without making bucket public
  • Allow logged-in user to upload directly to a specific S3 path
  • Temporary media access (e.g. video streaming for paying users)
Generation Method Max Expiration Notes
S3 Console 12 hours Quickest to generate, limited duration
AWS CLI / SDK 168 hours (7 days) Use --expires-in flag (seconds)
Architecture flow: App server generates pre-signed URL using its IAM role → Returns URL to client → Client uses URL directly against S3 → S3 validates signature and expiry → serves object. The app server credentials are never exposed to the client.
06 — S3 Access Points
S3 Access Points — Simplify Bucket Policies

Access Points solve the "bucket policy is getting too complex" problem. Instead of one massive bucket policy serving many teams, each team or application gets its own Access Point with its own scoped policy.

Access Point Properties
  • Each access point has its own unique DNS name
  • Each access point has its own IAM-like access policy
  • Policies scoped to specific prefixes (e.g., /finance/)
  • Simplifies management for large multi-tenant buckets
  • Can be scoped to VPC origin (private access only)
VPC Origin Access Points
  • Restricts access to traffic from a specific VPC
  • Traffic never leaves the AWS network
  • Requires a VPC Endpoint (Gateway or Interface) for S3
  • VPC Endpoint policy must allow access to the Access Point
  • Most secure pattern for internal team access

07 — S3 Object Lambda
S3 Object Lambda — Transform on Read

Object Lambda lets you run Lambda code to modify S3 objects as they are retrieved — without duplicating data into separate buckets for each transformation.
