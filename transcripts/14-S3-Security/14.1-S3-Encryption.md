# 14.1 - S3 Encryption

## Key Concepts

### 1. Four Encryption Methods

| Method | Key Managed By | Header | Notes |
|--------|---------------|--------|-------|
| **SSE-S3** | AWS (S3-owned key) | `"x-amz-server-side-encryption": "AES256"` | Default for new buckets/objects |
| **SSE-KMS** | You (via KMS service) | `"x-amz-server-side-encryption": "aws:kms"` | Audit key usage via CloudTrail |
| **SSE-C** | You (provide key in request) | Key in HTTP headers | HTTPS required, key never stored by AWS |
| **Client-Side** | You (encrypt before upload) | N/A | You encrypt/decrypt, S3 just stores the blob |

### 2. SSE-S3 (Default)

- Key is owned and managed by AWS — you never see it
- Encryption type: **AES-256**
- **Enabled by default** for all new buckets and objects
- Simplest option — no configuration needed

### 3. SSE-KMS

- You manage keys in **AWS KMS** (Key Management Service)
- Advantages:
  - **User control** over key creation and rotation
  - **CloudTrail** logs every key usage (audit trail)
- To read an object: need access to **both** the object AND the KMS key
- **Limitation:** KMS API calls count towards KMS quotas (5,000-30,000 req/sec per region)
  - High-throughput buckets may get **throttled** by KMS limits

### 4. SSE-C (Customer-Provided Keys)

- You send the encryption key **with every request** (in HTTP headers)
- AWS uses it, then **discards** it — never stores your key
- **HTTPS is mandatory** (you're transmitting a key)
- To read: must provide the same key again
- Not manageable from AWS Console — CLI/SDK only

### 5. Client-Side Encryption

- You encrypt data **before** uploading to S3
- You decrypt data **after** downloading from S3
- S3 just stores an encrypted blob — knows nothing about the encryption
- You fully manage keys and encryption cycle
- Can use AWS Client-Side Encryption Library

### 6. Encryption in Transit (SSL/TLS)

- S3 has two endpoints:
  - `http://` — not encrypted
  - `https://` — encrypted (SSL/TLS)
- **HTTPS recommended** for all operations
- **HTTPS required** for SSE-C
- Force HTTPS via bucket policy:

```json
{
  "Effect": "Deny",
  "Action": "s3:GetObject",
  "Condition": {
    "Bool": { "aws:SecureTransport": "false" }
  }
}
```

### 7. Default Encryption vs Bucket Policies

- All new buckets have **SSE-S3 by default** (can change to SSE-KMS)
- You can also **force encryption** via bucket policy — deny PUTs without correct encryption headers
- **Bucket policies are evaluated BEFORE default encryption settings**

```json
// Force SSE-KMS: deny if header is not aws:kms
{ "Effect": "Deny", "Action": "s3:PutObject",
  "Condition": { "StringNotEquals": { "s3:x-amz-server-side-encryption": "aws:kms" } } }

// Force SSE-C: deny if no customer algorithm header
{ "Effect": "Deny", "Action": "s3:PutObject",
  "Condition": { "Null": { "s3:x-amz-server-side-encryption-customer-algorithm": "true" } } }
```

---

## Exam Tips
- **SSE-S3** = default, AES-256, header `AES256`
- **SSE-KMS** = audit via CloudTrail, header `aws:kms`, watch for KMS throttling on high-throughput
- **SSE-C** = you provide key every request, HTTPS mandatory, key never stored
- **Client-Side** = you encrypt before upload, S3 stores encrypted blob
- Force HTTPS = bucket policy with `aws:SecureTransport` condition
- SSE-KMS throttling = 5,000-30,000 req/sec KMS quota per region
- Default encryption = SSE-S3, but bucket policies are evaluated **before** default encryption
- Bucket policy can force a specific encryption type (deny PUTs without correct headers)
