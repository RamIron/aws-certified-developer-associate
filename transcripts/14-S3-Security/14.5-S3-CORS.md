# 14.5 - S3 CORS

## Key Concepts

### 1. What is an Origin?

An origin = **scheme (protocol) + host (domain) + port**

```
https://www.example.com:443
  ↑          ↑            ↑
scheme     host          port
```

- Same origin: `https://www.example.com/page1` and `https://www.example.com/page2`
- Different origin: `https://www.example.com` and `https://other.example.com`

### 2. What is CORS?

- **Cross-Origin Resource Sharing**
- A **web browser** security mechanism
- Allows or denies requests from one origin to another
- Uses the `Access-Control-Allow-Origin` header

### 3. How CORS Works

```
1. Browser visits https://www.example.com → gets index.html
2. index.html says "load image from https://other.example.com"
3. Browser sends pre-flight request to other.example.com:
   "Hey, my origin is example.com — can I make requests?"
4. other.example.com responds with CORS headers:
   "Yes, I allow example.com for GET, PUT, DELETE"
5. Browser makes the actual request
```

### 4. CORS with S3

If a client makes a **cross-origin request** to an S3 bucket, you must enable CORS headers on the target bucket.

```
Browser → S3 Bucket A (static website, index.html)
               ↓ "load image from Bucket B"
        → S3 Bucket B (static website, assets)
               ↓ needs CORS headers allowing Bucket A's origin
        → Image returned ✅
```

- Configure CORS on the **target** bucket (the one being called)
- Allow a specific origin or `*` (all origins)

---

## Exam Tips
- CORS = **web browser** security (not server-side)
- If S3 static website needs assets from another S3 bucket → enable **CORS on the target bucket**
- CORS header: `Access-Control-Allow-Origin`
- Allow specific origin or `*` for all origins
- "Cross-origin request blocked on S3" → CORS not configured on the target bucket
