# 14.13 - S3 Access Points

## Key Concepts

### 1. The Problem

One S3 bucket with lots of data (finance, sales, analytics) and many users/groups. The bucket policy becomes huge and unmanageable.

### 2. The Solution: Access Points

Create separate **access points** for different use cases, each with its own policy:

```
S3 Bucket (finance + sales data)
    ├── Finance Access Point → policy: R/W to /finance prefix
    ├── Sales Access Point   → policy: R/W to /sales prefix
    └── Analytics Access Point → policy: READ to /finance + /sales
```

- Each access point has its own **DNS name** (how you connect)
- Each has its own **access point policy** (looks like a bucket policy)
- The bucket policy itself stays simple
- Scales access management easily

### 3. Access Point Origins

| Origin | Use Case |
|--------|----------|
| **Internet** | Public access via DNS name |
| **VPC** | Private access from within a VPC |

### 4. VPC Origin (Private Access)

```
EC2 Instance → VPC Endpoint → S3 Access Point (VPC origin) → S3 Bucket
```

- Requires a **VPC Endpoint** to connect privately to the access point
- VPC Endpoint has its own **policy** allowing access to the access point and bucket
- Three layers of security: VPC Endpoint policy + Access Point policy + Bucket policy

---

## Exam Tips
- Access Points = **simplify security** for S3 buckets with many users/data types
- Each access point has its own **DNS name** and **policy**
- Push security from one giant bucket policy into multiple smaller access point policies
- VPC origin = private access, requires **VPC Endpoint**
- "S3 bucket policy is too complex" → use Access Points
