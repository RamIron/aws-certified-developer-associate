What is Elastic Beanstalk?

👨‍💻

Developer-centric

You only worry about the code. Beanstalk handles provisioning, scaling, and monitoring.

🔧

Managed service

Reuses EC2, ASG, ELB, RDS, ElastiCache under the hood. You don't manage them directly.

💸

Free to use

Beanstalk itself costs nothing. You pay for the underlying resources (EC2, RDS, etc.).

🏗

Powered by CloudFormation

Each environment = one CloudFormation stack. Beanstalk creates and manages it for you.

Components

Application

Top-level container for all environments, versions, and configs.

App Version

An iteration of your code — v1, v2, v3… stored as a zip in S3.

Environment

A collection of AWS resources running one application version at a time.

Tier

Web Server (HTTP) or Worker (SQS background processing).

Lifecycle Policy

Beanstalk stores application versions in S3. The limit is 1000. Without cleanup, you eventually can't deploy.

Versions stored
LIMIT

At 1000 versions → new deployments are BLOCKED until old ones are removed.

🔢

By Count

Keep only the last N versions (e.g., keep last 200). Older ones are deleted.

📅

By Time

Delete versions older than N days (e.g., older than 180 days).

🛡

Active = Protected

Versions currently deployed to an environment are never deleted.

📦

S3 Source Bundle

Choose to retain (audit trail) or delete (save cost) the S3 zip file.

Lifecycle cleanup runs via a Beanstalk service role that needs IAM permissions to delete from S3.
4

EB Extensions

Every setting in the Beanstalk console can also be configured as code inside your source zip. This is infrastructure-as-code for Beanstalk.

Three Rules (all required)

📁

Directory

Must be in .ebextensions/ at the root of your source code. The dot matters.

📄

File extension

Files must end in .config (e.g., environment-variables.config, logging.config).

✏️

Format

Content must be YAML or JSON. The .config extension doesn't restrict format.

Example: Setting environment variables

# .ebextensions/environment-variables.config option_settings: aws:elasticbeanstalk:application:environment: DB_URL: "jdbc:postgresql://my-rds.amazonaws.com:5432/mydb" DB_USER: "admin" NODE_ENV: "production"
⚠ Resources created via EB Extensions (RDS, ElastiCache, DynamoDB) are tied to the environment. Delete the environment → those resources are deleted too.
5

CloudFormation Under the Hood

Beanstalk doesn't provision resources directly — it creates a CloudFormation stack for each environment. CloudFormation does the actual work.

🔍

1 Env = 1 Stack

Each Beanstalk environment maps to exactly one CloudFormation stack. Visible in the CF console.

⚙️

What it creates

ASG, Launch Config, ELB, Target Groups, Security Groups, CloudWatch Alarms, Scaling Policies.

🚀

Power unlock

EB Extensions + CloudFormation = provision any AWS resource beyond the console UI.

6

Cloning

Clone an existing environment to get an identical copy. Useful for creating test/staging from production in one click.

✓ Preserved

  • Load balancer type & config
  • RDS database configuration (type, size, settings)
  • All environment variables
  • Instance type & all resource settings
  • Application version

✗ NOT Preserved

  • RDS data (empty database)

Migrations

Load Balancer Migration

You cannot change the LB type after an environment is created — only its configuration. You also cannot use Clone for this: clone copies the LB type too.
1
Create a new environment manually with the same config but the desired LB type (e.g., Classic → ALB). Cannot clone — that copies the old LB type.
2
Deploy the application to the new environment.
3
Shift traffic via CNAME swap or Route 53 DNS update.
4
Terminate the old environment once traffic is fully migrated.

RDS Decoupling (from Beanstalk)

If RDS was created inside Beanstalk, it will be deleted with the environment. Here's how to safely decouple it:

1
Take an RDS snapshot — safety net in case anything goes wrong.
2
Enable RDS deletion protection in the RDS console. This prevents deletion no matter what.
3
Create a new Beanstalk environment — without RDS this time.
4
Point the new env to the existing RDS using an environment variable (DB connection string).
5
CNAME swap (or Route 53) to send all traffic to the new environment. Verify everything works.
6
Terminate the old environment. CloudFormation tries to delete RDS → blocked by deletion protection.
7
CloudFormation stack → DELETE_FAILED — this is expected and correct. Deletion protection worked. Go to CloudFormation and manually delete the stack.
✓ RDS now lives independently, connected to the new Beanstalk environment via env variable.
