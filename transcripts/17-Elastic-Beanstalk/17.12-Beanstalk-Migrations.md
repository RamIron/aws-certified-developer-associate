# 17.12 - Beanstalk Migrations

## Key Concepts

### 1. Load Balancer Migration

**The constraint:** After creating a Beanstalk environment, you **cannot change the Load Balancer type** — only its configuration.

**Example:** Can't upgrade Classic LB → ALB on the same environment.

**Migration steps:**
1. Create a **new environment** with same config but different LB type (can't use Clone — it copies the LB type too)
2. Deploy the application to the new environment
3. Shift traffic using **CNAME swap** or **Route 53 DNS update**
4. Terminate the old environment

```
Old env (Classic LB) → still running
New env (ALB)        → deploy app here
                     → CNAME swap / Route 53 update
                     → terminate old env
```

> ⚠️ **Cannot use Clone** — clone copies the exact LB type and config

---

### 2. Decoupling RDS from Beanstalk (Migration)

**The problem:** RDS inside Beanstalk = database deleted when environment is deleted. Bad for production.

**How to decouple an existing RDS from Beanstalk:**

| Step | Action |
|------|--------|
| 1 | Take a **snapshot** of the RDS database (safeguard) |
| 2 | Enable **RDS deletion protection** in the RDS console |
| 3 | Create a **new Beanstalk environment without RDS** |
| 4 | Point the new environment to the existing RDS via **environment variable** (connection string) |
| 5 | Perform **CNAME swap** or Route 53 update to shift traffic |
| 6 | **Terminate the old environment** |
| 7 | RDS survives because of deletion protection |
| 8 | The CloudFormation stack will be in **DELETE_FAILED** state — delete it manually |

```
Old env (Beanstalk + RDS inside)
        ↓ snapshot RDS + enable deletion protection
New env (Beanstalk only) → points to RDS via env var
        ↓ CNAME swap
Old env terminated → CloudFormation tries to delete RDS → FAILS (deletion protection)
        ↓ manually delete CloudFormation stack
RDS now lives independently ✓
```

---

## Exam Tips

- **Cannot change LB type** after environment creation — must migrate
- **Cannot use Clone** for LB migration — clone copies the LB type
- Use **CNAME swap or Route 53** to shift traffic during migrations
- **RDS inside Beanstalk** = deleted with environment → decouple for production
- Decoupling RDS: snapshot → deletion protection → new env → CNAME swap → terminate old → manually delete CloudFormation stack
- After terminating old env with protected RDS: CloudFormation stack will be in **DELETE_FAILED** — delete it manually
