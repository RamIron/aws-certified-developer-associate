# 17.5 - Beanstalk Deployment Modes

## Key Concepts

### Summary Table

| Mode | Downtime? | Capacity during deploy | Cost | Rollback | Speed |
|------|-----------|----------------------|------|----------|-------|
| **All at once** | YES | Zero (all stopped) | No extra | Redeploy old version | Fastest |
| **Rolling** | No | Below capacity | No extra | Manual (redeploy) | Slow (depends on bucket size) |
| **Rolling + extra batch** | No | At capacity (or above) | Small extra (extra batch) | Manual (redeploy) | Slow |
| **Immutable** | No | Double capacity (new temp ASG) | High (double instances) | Quick (terminate temp ASG) | Longest |
| **Blue/Green** | No | Double (2 environments) | High (2 environments) | Quick (swap URL back) | Manual |
| **Traffic Splitting** | No | Double capacity (temp ASG) | High (double instances) | Automatic (stop traffic to temp ASG) | Automated |

### 1. All at Once

```
v1 v1 v1 v1  →  __ __ __ __  →  v2 v2 v2 v2
(all running)   (all stopped)   (all updated)
```

- Fastest deployment
- **Downtime** (all instances stop at once)
- No additional cost
- Good for: **development** (quick iterations, don't care about downtime)

### 2. Rolling

```
v1 v1 v1 v1  →  __ __ v1 v1  →  v2 v2 v1 v1  →  v2 v2 __ __  →  v2 v2 v2 v2
                (bucket stops)   (bucket updated)  (next bucket)   (all done)
```

- **Below capacity** during deployment (bucket size instances are stopped)
- No additional cost
- Both versions running simultaneously during deploy
- Long deployment if small bucket size + many instances

### 3. Rolling with Additional Batches

```
v1 v1 v1 v1  →  v1 v1 v1 v1 v2 v2  →  v1 v1 v2 v2 v2 v2  →  v2 v2 v2 v2
                (extra batch added)     (first bucket updated)   (extra removed)
```

- **At capacity** throughout (new instances created first)
- Small additional cost (extra batch)
- Extra batch removed at the end
- Good for: **production** (never below capacity)

### 4. Immutable

```
Current ASG:  v1 v1 v1
Temp ASG:     v2 (test 1 first)  →  v2 v2 v2 (all healthy)
Merge:        v1 v1 v1 v2 v2 v2  →  v2 v2 v2 (terminate old)
Temp ASG:     (removed)
```

- Creates a **temporary ASG** with all new instances
- First deploys 1 instance to test, then the rest
- **Double capacity** during deployment (high cost)
- **Quick rollback** = just terminate temp ASG
- Longest deployment
- Great for: **production** (willing to pay for safety)

### 5. Blue/Green (Manual, not a direct Beanstalk feature)

```
Blue Environment (v1):   ELB → ASG → v1 v1 v1  ← 90% traffic (Route 53)
Green Environment (v2):  ELB → ASG → v2 v2 v2  ← 10% traffic (Route 53)

When happy → swap URLs → green becomes primary
```

- Create a **separate Beanstalk environment** for v2
- Use **Route 53 weighted policy** to split traffic
- Very **manual** process (not automated by Beanstalk)
- Swap URLs when ready

### 6. Traffic Splitting (Canary Testing)

```
Main ASG (v1):  v1 v1 v1  ← 90% traffic (via ALB)
Temp ASG (v2):  v2 v2 v2  ← 10% traffic (via ALB)

Health monitored → if failure → automatic rollback
If healthy → migrate v2 instances to main ASG → terminate v1
```

- Like Blue/Green but **automated** by Beanstalk
- Uses **ALB** to split traffic (not Route 53)
- **Automatic rollback** on failure
- "Canary testing" in the exam = **Traffic Splitting**

### AWS Official Comparison Table

Source: [AWS Docs](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/using-features.deploy-existing-version.html)

| Method | Impact of failed deployment | Deploy time | Zero downtime | No DNS change | Rollback process | Code deployed to |
|--------|---------------------------|-------------|--------------|---------------|-----------------|-----------------|
| **All at once** | Downtime | Fastest | No | Yes | Manual redeploy | Existing instances |
| **Rolling** | Single batch out of service; successful batches run new version | Slow † | Yes | Yes | Manual redeploy | Existing instances |
| **Rolling + extra batch** | Minimal if first batch fails; otherwise similar to Rolling | Slower † | Yes | Yes | Manual redeploy | New and existing instances |
| **Immutable** | Minimal | Slowest | Yes | Yes | Terminate new instances | New instances |
| **Traffic splitting** | % of traffic temporarily impacted | Slowest †† | Yes | Yes | Reroute traffic + terminate new instances | New instances |
| **Blue/Green** | Minimal | Slowest | Yes | **No** | Swap URL | New instances |

† Varies depending on batch size. †† Varies depending on evaluation time setting.

---

## Exam Tips
- **All at once** = only mode with **downtime**, fastest, dev only
- **Rolling** = below capacity, no extra cost
- **Rolling + extra batch** = at capacity, small extra cost, good for prod
- **Immutable** = temp ASG, double capacity, quick rollback, longest deploy
- **Blue/Green** = separate environment, Route 53 weighted, **manual**
- **Traffic Splitting** = temp ASG, ALB splits traffic, **automated**, canary testing
- "Canary testing" = **Traffic Splitting**
- "No downtime + no extra cost" = **Rolling**
- "No downtime + always at capacity + minimal cost" = **Rolling with additional batches**
- "Fastest rollback" = **Immutable** or **Traffic Splitting**
- "Zero downtime + quick rollback + willing to pay" = **Immutable**
