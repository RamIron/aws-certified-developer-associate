# 17.8 - Beanstalk Lifecycle Policy

## Key Concepts

### 1. The Problem: 1000 Version Limit

- Beanstalk can store up to **1000 application versions**
- If you don't delete old versions, you'll eventually hit the limit
- At 1000 versions, you **cannot deploy new versions** until old ones are removed
- Solution: **Lifecycle Policy** to automatically phase out old versions

### 2. Lifecycle Policy Options

Two ways to phase out old versions:

| Policy Type | Description |
|-------------|-------------|
| **By count** | Keep max N versions (e.g., keep last 200) |
| **By time** | Delete versions older than N days (e.g., older than 180 days) |

### 3. Safety: Active Versions Not Deleted

- Versions **currently deployed** to an environment are **never deleted** by the lifecycle policy
- This protects running environments from accidental deletion

### 4. S3 Source Bundle Option

- Application versions are stored in **S3 behind the scenes**
- Lifecycle policy can be configured to:
  - **Delete** the source bundle in S3 (saves storage cost)
  - **Retain** the source bundle in S3 (prevents data loss, keeps audit trail)

### 5. Service Role Requirement

- The lifecycle policy is executed by Beanstalk using a **service role**
- The service role must have **IAM permissions** to delete versions from S3

---

## Exam Tips

- **1000 version limit** → use Lifecycle Policy to avoid hitting it
- Two phase-out options: **count** (max N versions) or **time** (older than N days)
- **Active versions** (deployed to an environment) are **never deleted**
- Option to retain or delete the **S3 source bundle**
- Lifecycle policy uses a **Beanstalk service role** with IAM permissions
