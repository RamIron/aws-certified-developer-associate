01 — EBS (Elastic Block Store)
Network Drive
Not a physical drive — communicates over the network. Slight latency possible. Can be detached from one EC2 and attached to another.
AZ-Locked
Cannot attach to an instance in a different AZ. An EBS volume in us-east-1a cannot be directly attached to an instance in us-east-1b.
Persistent Storage
Data survives instance termination. This is the key advantage over EC2 Instance Store (which is ephemeral).
Delete on Termination
Root volume: deleted by default when instance terminates.
Additional volumes: kept by default (can be changed).
02 — EBS Snapshots
Point-in-time backup of an EBS volume stored in S3 (managed by AWS)
Not required to detach the volume first, but recommended for data consistency
Use snapshots to MIGRATE EBS volumes across AZs or Regions — snapshot in AZ-A, restore in AZ-B
Snapshot Features
Snapshot Archive
75% cheaper storage tier for snapshots that are rarely accessed. Restore time takes 24 to 72 hours.
Recycle Bin
Recover accidentally deleted snapshots. Configure retention rules from 1 day to 1 year.
Fast Snapshot Restore
Forces full initialization of a snapshot — eliminates the latency on first use. Costs more.
03 — AMI (Amazon Machine Image)
Customization of EC2 — pre-packages your OS, software, configuration, and monitoring agents. Faster boot since software is pre-baked.
Region-specific — AMIs are built for a specific region. Can be copied across regions but cannot be used directly across regions.
AMI Types
AWS PROVIDED
Public AMI
Provided and maintained by AWS. Example: Amazon Linux 2, Ubuntu, Windows Server. Free to use.
YOUR OWN
Custom AMI
Created and maintained by you. Contains your custom software, config, packages. Faster launch times.
THIRD PARTY
Marketplace AMI
AMIs sold by third-party vendors on the AWS Marketplace. May include licensed software with cost.
AMI Creation Flow
1
Launch + customize an EC2 instance — install software, configure as needed
2
Stop the instance — ensures data integrity before imaging
3
Build the AMI — AWS creates EBS snapshots of all volumes behind the scenes
4
Launch new instances from the AMI — instantly provisioned with all pre-packaged software
04 — EC2 Instance Store
Physical hard drive attached directly to the HOST hardware — not a network drive
FASTEST storage on AWS — millions of IOPS vs thousands for EBS. Best raw throughput.
EPHEMERAL — data is LOST when the instance is stopped, terminated, or fails
Good for: buffer, cache, scratch data, temporary content — anything stateless and replaceable
Your responsibility to backup and replicate — AWS does not manage persistence for Instance Store

SSD

Boot volumeYes

SSD
Max IOPS256,000 (Block Express)
Throughput7,500 MiB/s
Boot volumeYes
Best for: Same as io1 but higher durability (99.999%). io2 Block Express reaches 256K IOPS. Supports Multi-Attach.
st1
HDD
Max IOPS500
Throughput500 MiB/s
Boot volumeNo
Best for: Throughput-optimized HDD. Big Data, data warehouses, log processing. High sequential throughput.
sc1
HDD
Max IOPS250
Throughput250 MiB/s
Boot volumeNo
Best for: Cold HDD — lowest cost. Infrequently accessed data where cost matters most. Archive workloads.
gp3 vs gp2: gp3 IOPS is independent from size — you can provision more IOPS without buying a larger volume. gp2 IOPS is tied to size: 3 IOPS/GiB, maxing at 16,000 IOPS at 5,334 GiB.
Boot volumes: Only SSD types can be used as root/boot volumes (gp2, gp3, io1, io2). HDD types (st1, sc1) cannot boot.
IOPS thresholds: Need >16K IOPS → use io1 or io2. Need >256K IOPS → only EC2 Instance Store can exceed io2 Block Express's 256K max.
06 — EBS Multi-Attach
Supported Types
io1 and io2 ONLY. General purpose (gp2, gp3) and HDD types do not support Multi-Attach.
AZ Constraint
Same AZ only. You cannot Multi-Attach across Availability Zones.
Instance Limit
Maximum 16 EC2 instances can be attached to a single Multi-Attach volume simultaneously.
File System
Requires a cluster-aware file system. NOT compatible with XFS or EXT4 — use GFS2 or OCFS2 for concurrent writes.
07 — Amazon EFS (Elastic File System)
Managed NFS
Network File System (NFS protocol). Fully managed — no provisioning, patching, or capacity planning required.
Linux Only
POSIX file system — Linux only. Does NOT support Windows instances. Uses standard NFS file permissions.
Scale
Mount across hundreds of EC2 instances in multiple AZs simultaneously. Truly shared storage.
Cost & Port
~3x more expensive than gp2. But pay-per-use — no capacity to provision. Port 2049 (NFS).
Performance Modes (set at creation, cannot change)
Mode Latency Throughput Best For
General Purpose Low latency Normal Default — web servers, CMS, home directories, development
Max I/O Higher latency Very high Big data analytics, media processing, high parallelism workloads
Throughput Modes
Mode Scaling Behavior Best For
Bursting Scales with storage size Variable workloads, typical NFS use cases
Provisioned Set independently of storage size Known high-throughput workloads (e.g., 1 GiB/s for 1 TB stored)
Elastic Auto-scales up/down (up to 3 GiB/s reads) Unpredictable workloads — recommended default in modern AWS
Storage Tiers (Lifecycle Management)
Tier Access Pattern Cost
Standard Frequently accessed Higher storage cost
EFS-IA Infrequently accessed Lower storage cost + retrieval fee per access
Archive Rarely accessed (a few times per year) Cheapest storage cost
Weak Point Trap Performance Modes vs Throughput Modes — do NOT confuse these:

Performance Mode = about latency + parallelism (General Purpose = low latency / Max I/O = massive throughput but higher latency)
Throughput Mode = about how much data can flow (Bursting = tied to storage / Provisioned = fixed / Elastic = auto-scales)

General Purpose does NOT mean low throughput. Max I/O does NOT mean "more expensive throughput mode."

09 — Exam Tips
Fastest storage option on AWS?
EC2 Instance Store — millions of IOPS, but ephemeral
AZ-locked network block storage?
EBS — cannot be directly used across AZs
Move EBS to another AZ?
Take a snapshot, then restore in the new AZ
Boot volumes must be...
SSD only (gp2, gp3, io1, io2) — HDD cannot boot
gp3 vs gp2 key difference?
gp3 IOPS independent from size; gp2 IOPS tied to size (3/GiB)
Need more than 256K IOPS?
Only EC2 Instance Store — io2 Block Express maxes at 256K
Multi-Attach allowed for?
io1 and io2 only — same AZ, max 16 instances, cluster-aware FS required
EFS is ___ compatible
Linux only (POSIX) — does NOT work on Windows
EFS works across AZs?
Yes — can be mounted on instances across multiple AZs simultaneously
Cheapest EFS option?
One Zone-IA — single AZ + infrequent access tier
Shared file system for hundreds of EC2s?
EFS — the only option that supports concurrent access from hundreds of instances
EFS performance mode for big data?
Max I/O (not General Purpose) — higher latency but massive parallel throughput
EFS throughput for unpredictable workloads?
Elastic — auto-scales throughput up and down as needed
