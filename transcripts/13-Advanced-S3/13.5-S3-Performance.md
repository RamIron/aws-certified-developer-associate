# 13.5 - S3 Performance

## Key Concepts

### 1. Baseline Performance

- Latency: **100-200 ms** to get the first byte
- **3,500 PUT/COPY/POST/DELETE** per second per prefix
- **5,500 GET/HEAD** per second per prefix
- No limit to the number of prefixes in a bucket

### 2. What is a Prefix?

The prefix is everything between the bucket name and the file name:

```
s3://my-bucket/folder1/sub1/file  →  prefix: /folder1/sub1/
s3://my-bucket/folder1/sub2/file  →  prefix: /folder1/sub2/
s3://my-bucket/folder2/sub1/file  →  prefix: /folder2/sub1/
s3://my-bucket/folder2/sub2/file  →  prefix: /folder2/sub2/
```

4 prefixes × 5,500 GET = **22,000 GET/s** by spreading reads evenly.

### 3. Optimizing Uploads — Multi-Part Upload

- **Recommended** for files > 100 MB
- **Required** for files > 5 GB
- Splits the file into parts, uploads them **in parallel**
- S3 reassembles the parts into the original file

### 4. Optimizing Uploads — S3 Transfer Acceleration

- Upload to a nearby **AWS edge location** (200+ worldwide)
- Edge location forwards data over **fast private AWS network** to the target S3 bucket
- Minimizes public internet, maximizes private network
- Compatible with multi-part upload

```
USA (file) → Edge Location (USA, public internet) → S3 Bucket (Australia, private AWS network)
```

### 5. Optimizing Downloads — S3 Byte-Range Fetches

- Request specific **byte ranges** of a file in parallel
- Speeds up downloads by parallelizing GETs
- Better resilience: if one range fails, retry just that small part
- Can also retrieve **partial data** (e.g., first 50 bytes for a file header)

---

## Exam Tips
- **3,500 PUT** / **5,500 GET** per second **per prefix**
- Spread reads across prefixes to multiply performance
- Upload optimization: **Multi-Part Upload** (parallel parts) + **Transfer Acceleration** (edge locations)
- Download optimization: **Byte-Range Fetches** (parallel partial GETs)
- Multi-Part: recommended > 100 MB, required > 5 GB
- Transfer Acceleration: public internet → edge location → private AWS network
