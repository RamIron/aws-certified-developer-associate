# 13.3 - S3 Event Notifications

## Key Concepts

### 1. What Are S3 Events?

Events are things that happen in your bucket:
- Object **created**
- Object **removed**
- Object **restored**
- **Replication** happened

You can **filter** events (e.g., only objects ending with `.jpg`).

### 2. Event Notification Destinations

| Destination | Permission Needed |
|-------------|------------------|
| **SNS Topic** | SNS Resource Access Policy |
| **SQS Queue** | SQS Resource Access Policy |
| **Lambda Function** | Lambda Resource Policy |

- These use **resource-based policies** (not IAM Roles for S3)
- Works similarly to S3 Bucket Policies — you grant S3 permission to send to the target

### 3. Delivery

- Typically delivered within **seconds**
- Can sometimes take a **minute or longer**
- You can create **as many** event notifications as desired

### 4. Amazon EventBridge Integration

```
S3 Events → ALL events go to EventBridge automatically
                    ↓
              EventBridge Rules → 18+ AWS services as destinations
```

EventBridge advantages over direct SNS/SQS/Lambda:
- **Advanced filtering** — by metadata, object size, name
- **Multiple destinations** at once
- **Archive & replay** events
- **More reliable** delivery
- Can send to Step Functions, Kinesis Data Streams, Firehose, etc.

---

## Exam Tips
- S3 Event Notification targets: **SNS, SQS, Lambda, EventBridge**
- SNS/SQS/Lambda require **resource-based policies** (not IAM Roles)
- EventBridge receives ALL S3 events → can route to 18+ services
- EventBridge = advanced filtering + multiple destinations + archive/replay
- Use case: "React to S3 uploads automatically" → Event Notifications
