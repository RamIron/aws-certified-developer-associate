1 · RDS Overview
Amazon RDS — Relational Database Service

Fully managed relational DB service. AWS handles OS patching, backups, monitoring, and scaling. You cannot SSH into the underlying EC2 instance (except RDS Custom). Storage is EBS-backed with Storage Auto Scaling.

2 · Read Replicas vs Multi-AZ
Feature Comparison
Feature Read Replicas Multi-AZ

Storage Architecture
  • 6 copies across 3 Availability Zones
  • 4 out of 6 copies needed for writes
  • 3 out of 6 copies needed for reads
  • Self-healing peer-to-peer replication
  • Storage is striped across 100s of volumes
Endpoints
  • Writer Endpoint — always points to master
  • Reader Endpoint — load-balanced across all read replicas
  • Up to 15 read replicas
  • Replica lag under 10ms
  • ~20% more expensive than standard RDS
Backtrack: Restore the Aurora DB to any point in time without relying on backups. Rewind the database in-place — no need to create a new cluster. Configured at cluster creation time.
4 · RDS & Aurora Security
Security Layers
LayerDetail
At-Rest Encryption KMS-AES-256. Must be defined at launch. Unencrypted master = unencrypted replicas.
In-Flight Encryption TLS enabled by default for all connections
Authentication Username/password or IAM roles (for MySQL & PostgreSQL)
Network Security Groups control inbound/outbound traffic
SSH Access Not available — except RDS Custom
Audit Logs Can be enabled → forwarded to CloudWatch Logs for long-term retention
Encrypting an existing unencrypted DB: You cannot enable encryption on a running instance. The process is: take a snapshot of the unencrypted DB → restore the snapshot as a new encrypted DB → update connection strings.

6 · ElastiCache — Redis vs Memcached
Managed In-Memory Caching
Feature Redis Memcached
High Availability Multi-AZ auto-failover No HA
Read Scaling Read replicas No replicas
Data Durability AOF persistence No persistence
Backup & Restore Yes Serverless version only
Data Structures Sets, sorted sets, hashes, lists, streams Simple key-value only
Architecture Replication (primary + replicas) Sharding (horizontal partitioning)
Multi-threaded No Yes
Best For Leaderboards, sessions, pub/sub, durable cache Simple caching, high throughput, multi-thread
Key Reminder: ElastiCache requires application code changes to work. The application must be programmed to check the cache before hitting the DB. IAM auth is NOT supported in ElastiCache — use Redis AUTH or Memcached SASL.

8 · MemoryDB for Redis

Use MemoryDB when: You need a durable, Redis-compatible primary database (not just a cache) with ultra-fast performance. It stores the entire dataset in memory AND persists using a Multi-AZ transaction log.
ElastiCache vs MemoryDB: ElastiCache is a cache in front of a DB. MemoryDB replaces the DB entirely. If the question says "durable in-memory database," the answer is MemoryDB.
