# 16.8 - Amazon ECS Task Definitions - Deep Dive

## Key Concepts

### 1. What's in a Task Definition?

Defined in **JSON** (console has a UI helper). Tells ECS how to run containers:
- **Image name** (which Docker image to use)
- **Port bindings** (container port + host port for EC2)
- **CPU and memory** requirements
- **Environment variables**
- **IAM Role** (Task Role)
- **Logging configuration** (e.g., CloudWatch)
- **Networking** info
- Up to **10 containers** per task definition
- At least one must be **essential** (if it stops → entire task stops)

### 2. Port Mapping: EC2 vs Fargate

#### EC2 Launch Type — Dynamic Host Port Mapping

```
EC2 Instance
  ├── Task 1: container port 80 → host port 32768 (random)
  ├── Task 2: container port 80 → host port 32769 (random)
  └── Task 3: container port 80 → host port 32770 (random)
       ↑
  ALB knows all these ports automatically (via ECS service)
```

- Set host port to **0** (or don't set it) → Docker assigns a random port
- **ALB** can handle this (finds the right port automatically). **CLB cannot.**
- Security Group: EC2 must allow **any port** from ALB security group

#### Fargate Launch Type — Unique Private IP per Task

```
ECS Cluster (Fargate)
  ├── Task 1: 10.0.0.5:80 (own ENI)
  ├── Task 2: 10.0.0.6:80 (own ENI)
  ├── Task 3: 10.0.0.7:80 (own ENI)
  └── Task 4: 10.0.0.8:80 (own ENI)
       ↑
  ALB connects to all on port 80
```

- Each task gets its own **private IP** via an **ENI** (Elastic Network Interface)
- No host port needed — only define **container port**
- Security Group: ENI must allow **port 80** from ALB security group

### 3. IAM Roles: Defined at Task Definition Level

```
Task Definition A (Task Role A → access S3)
  └── ECS Service → Task 1, Task 2, Task 3 → all get Task Role A

Task Definition B (Task Role B → access DynamoDB)
  └── ECS Service → Task 4, Task 5 → all get Task Role B
```

- Role is defined on the **task definition**, NOT on the service
- All tasks from that definition inherit the same role

### 4. Environment Variables

| Source | Use case |
|--------|----------|
| **Hardcoded** | Fixed, non-secret values (e.g., a URL) |
| **SSM Parameter Store** | Sensitive or shared config values |
| **Secrets Manager** | Secrets (API keys, DB passwords) |
| **S3 bucket** | Bulk environment variables from a file |

- SSM/Secrets Manager values are **fetched at runtime** and injected into the container

### 5. Data Sharing Between Containers (Bind Mounts)

Multiple containers in the same task definition can share data via **bind mounts**:

```
ECS Task
  ├── App Container → writes to /var/logs ──┐
  │                                          ├── Bind Mount (shared storage)
  └── Sidecar Container → reads /var/logs ──┘
      (sends logs/metrics to external service)
```

- Works with **both** EC2 and Fargate
- **EC2**: storage tied to EC2 instance lifecycle
- **Fargate**: ephemeral storage (20-200 GB), tied to task lifecycle — task dies = data gone
- Use case: **sidecar pattern** — app writes logs, sidecar ships them

---

## Exam Tips
- Task definition = **JSON blueprint** with image, ports, CPU/RAM, IAM role, env vars
- EC2: **Dynamic Host Port Mapping** (host port = 0, ALB finds it, CLB can't)
- Fargate: each task gets **unique private IP** via ENI, only container port needed
- IAM Role defined at **task definition** level (not service level)
- Secrets → **SSM Parameter Store** or **Secrets Manager** (never hardcode)
- Bulk env vars → load from **S3 file**
- Sidecar containers share data via **bind mounts**
- Fargate ephemeral storage: **20-200 GB**
