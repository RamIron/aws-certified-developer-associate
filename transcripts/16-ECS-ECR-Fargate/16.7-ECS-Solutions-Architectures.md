# 16.7 - Amazon ECS - Solutions Architectures

## Key Concepts

### 1. ECS Task Invoked by EventBridge (Event-Driven)

```
User uploads to S3 → EventBridge rule → runs ECS Task (Fargate)
                                          ├── Gets object from S3
                                          ├── Processes it
                                          └── Saves result to DynamoDB
                                          (uses ECS Task Role for permissions)
```

- Fully serverless (S3 + EventBridge + Fargate)
- Use case: process uploaded files (images, documents, etc.)

### 2. ECS Task on a Schedule (EventBridge Schedule)

```
EventBridge Schedule (every 1 hour) → runs ECS Task (Fargate)
                                        └── Batch processing on S3 files
                                        (uses ECS Task Role for S3 access)
```

- Fully serverless
- Use case: scheduled batch jobs (like a cron job but serverless)

### 3. ECS Service with SQS Queue (Queue-Driven)

```
Messages → SQS Queue → ECS Service (polls for messages)
                         ├── Task 1
                         ├── Task 2
                         └── Task 3 (auto-scaled based on queue depth)
```

- More messages in queue → Auto Scaling adds more tasks
- Use case: async processing, decoupled workloads

### 4. ECS Task Lifecycle Events (Intercept with EventBridge)

```
ECS Task stops/starts → EventBridge event → SNS → email to admin
```

- React to container lifecycle changes (stopped, started, failed)
- Use case: alerting when tasks crash or exit unexpectedly

---

## Exam Tips
- "Process S3 uploads with Docker" → **EventBridge + ECS Fargate task**
- "Run a batch job every hour with Docker" → **EventBridge Schedule + ECS Fargate task**
- "Process messages from a queue with containers" → **SQS + ECS Service + Auto Scaling**
- "Alert when ECS task crashes" → **EventBridge + SNS**
- All these patterns are **fully serverless** when using Fargate
