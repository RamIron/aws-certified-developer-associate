3 — ECS Launch Types
EC2 Launch Type vs Fargate
EC2 Launch Type
  • You provision & manage EC2 instances
  • ECS Agent must run on each instance
  • Task scaling + EC2 instance scaling
  • More control, more operational overhead
  • Supports GPU workloads
  • Cost optimized with Reserved/Spot EC2
Fargate Launch Type
  • AWS manages the underlying infrastructure
  • Serverless — no EC2 instances to manage
  • Task scaling only (no EC2 scaling needed)
  • Preferred on the exam for simplicity
  • Each task gets its own unique ENI
  • Pay per task vCPU & memory
ECS Hierarchy
ECS Cluster
↓
ECS Service (desired count)
↓
ECS Task (running containers)
Service maintains a desired count of tasks running at all times. If a task fails, the service replaces it automatically. A task runs one or more containers defined by a task definition.
4 — IAM Roles for ECS
Three Critical IAM Roles
EC2 Instance Profile
EC2 Launch Type ONLY
  • Register instance with ECS cluster
  • Pull images from ECR
  • Write logs to CloudWatch
  • Used by the ECS Agent on the host
ECS Task Role
Both EC2 & Fargate
  • Attached to individual tasks
  • Access S3, DynamoDB, RDS, etc.
  • Different tasks can have different roles
  • Defined in task definition JSON
Task Execution Role
Both EC2 & Fargate
  • Pull images from ECR
  • Fetch secrets from SSM / Secrets Manager
  • Write logs to CloudWatch
  • Used during task startup (bootstrap)
Hotel Analogy: The EC2 Instance Profile is the hotel manager — runs the building infrastructure. ECS Task Role is the guest key card — grants access to specific rooms (AWS services) per guest (task). Task Execution Role is the concierge — handles setup tasks like fetching secrets and pulling images.
5 — ECS Task Definitions Deep Dive
Task Definition — JSON Blueprint
Core Fields
  • Docker image URL
  • CPU & Memory allocation
  • Port mappings
  • IAM Task Role
  • Environment variables
  • Logging configuration
Port Mapping

EC2 Launch Type: Dynamic Host Port — set host port to 0, ALB auto-discovers. Allows multiple tasks per instance.

Fargate: Unique ENI per task. No host port needed — container port only.

Container Limits

Up to 10 containers per task definition.

At least 1 essential container required — if it stops, the whole task stops.

Environment Variables
  • Hardcoded (not recommended for secrets)
  • SSM Parameter Store reference
  • Secrets Manager reference
  • S3 file (bulk env vars)
Sidecar Pattern

Multiple containers in a task share a bind mount. Sidecar reads/writes from the same volume as the main app container — no networking needed between them.

Logging

Configure awslogs driver to send container stdout/stderr directly to CloudWatch Logs. Requires logs:CreateLogStream permission.

6 — ECS Rolling Updates
Rolling Deployment — Min/Max Capacity Controls

Rolling updates use Minimum healthy percent and Maximum percent to control how many tasks can be running during a deployment.

Scenario A — Min 50%, Max 100%
Can drop to 50% capacity. Cannot exceed 100%. Terminates first, then creates.
Scenario B — Min 100%, Max 150%
Must keep 100% running at all times. Can overprovision to 150%. Creates first, then terminates. Zero capacity loss.

Scaling Policies
Target Tracking

Set a target value for a metric (e.g., CPU = 60%). ECS auto-adjusts task count to maintain that target.

Step Scaling

Scale by specific amounts based on CloudWatch Alarm thresholds. More granular control than target tracking.

Scheduled Scaling

Pre-scale for known traffic patterns (e.g., every Monday morning). Anticipate demand before it arrives.

EC2 Launch Type — Cluster Capacity Provider: Auto Scaling on ECS tasks is not enough — you also need more EC2 instances. Use an ECS Cluster Capacity Provider (backed by an ASG) to automatically provision EC2 capacity when tasks can't be placed. A plain ASG alone is not sufficient.

11 — Amazon EKS
ECS vs EKS — When to use which
Amazon ECS
  • AWS-native, proprietary orchestration
  • Unit of work: ECS Task (container)
  • Tight AWS service integration
  • Simpler to set up and operate
  • Exam default for "least overhead"
Amazon EKS
  • Managed Kubernetes — open-source standard
  • Unit of work: EKS Pod (group of containers)
  • Cloud-agnostic, works across cloud providers
  • Use when: already running Kubernetes
  • Use when: migrating from on-prem K8s
EKS Node Types
Managed Node Groups

AWS creates and manages EC2 nodes (part of an ASG). Nodes are tagged for Kubernetes automatically.

Self-Managed Nodes

You create, register, and manage EC2 nodes. Use custom AMIs. More control but more ops overhead.

Fargate Mode

No nodes at all — AWS runs each pod serverlessly. Great for bursty, unpredictable workloads.

EKS + Fargate Storage: When using EKS with Fargate, the ONLY supported persistent storage option is Amazon EFS. EBS volumes are not supported with Fargate pods.
12 — Data Persistence
Storage Options for ECS Tasks
Amazon EFS

Network file system — shared, persistent storage.

  • Persistent across task restarts
  • Shared across multiple tasks simultaneously
  • Works across multiple AZs
  • Compatible with EC2 and Fargate launch types
  • EKS + Fargate: EFS is the ONLY persistent option
Fargate + EFS = Fully Serverless — serverless compute + serverless storage
Bind Mounts

Ephemeral volume shared within a single task.

  • Temporary — lost when task stops
  • Shared between containers in the same task
  • Works with EC2 and Fargate
  • Perfect for sidecar pattern
  • No cross-task or cross-AZ sharing
Sidecar sharing data: Main app writes to bind mount, sidecar reads it. No network call needed.
13 — Load Balancer Integration
ECS + Load Balancer Options
Load Balancer Supported EC2 Launch Fargate Notes
ALB — Application Recommended Yes Yes Dynamic port mapping, path-based routing, best for HTTP/HTTPS
NLB — Network Specific use Yes Yes High throughput, TCP/UDP, AWS PrivateLink only
CLB — Classic Not recommended Limited No Legacy. Does NOT work with Fargate. Avoid for new architectures.
Dynamic Port Mapping (EC2): Set the host port to 0 in the task definition. ECS picks a random available port. The ALB queries ECS to discover the actual port for each task — allowing multiple tasks of the same type on one instance.
