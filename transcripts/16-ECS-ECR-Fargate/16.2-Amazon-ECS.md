# 16.2 - Amazon ECS

## Key Concepts

### 1. EC2 Launch Type

- ECS Cluster is made of **EC2 instances** you provision and manage
- Each EC2 instance runs the **ECS Agent** (registers the instance into the cluster)
- When you start ECS tasks, AWS places containers on your EC2 instances automatically

```
ECS Cluster
  ├── EC2 Instance (ECS Agent) → Container A, Container B
  ├── EC2 Instance (ECS Agent) → Container A, Container C
  └── EC2 Instance (ECS Agent) → Container B
```

- **You** manage the infrastructure (patching, scaling EC2 instances)

### 2. Fargate Launch Type (Serverless)

- **No EC2 instances to manage** — it's serverless
- You create a **task definition** (CPU, RAM requirements)
- AWS runs the tasks for you — you don't know or care where
- To scale: just increase the number of tasks

```
ECS Cluster (Fargate)
  ├── Task A (1 vCPU, 2GB)
  ├── Task A (1 vCPU, 2GB)
  └── Task B (0.5 vCPU, 1GB)
  (no EC2 instances visible)
```

- **Exam loves Fargate** — serverless = easier to manage

### 3. IAM Roles for ECS

| Role | Applies to | Purpose |
|------|-----------|---------|
| **EC2 Instance Profile** | EC2 Launch Type only | Used by the **ECS Agent** to call ECS API, CloudWatch Logs, ECR (pull images), Secrets Manager/SSM |
| **ECS Task Role** | Both EC2 and Fargate | Permissions for the **container itself** (e.g., access S3, DynamoDB) |

**Full picture (EC2 Launch Type):**
```
ECS Cluster
│
├── EC2 Instance #1
│   ├── ECS Agent (background program, always running)
│   │   └── Uses: Instance Profile (IAM Role)
│   │       - Registers EC2 into the cluster
│   │       - Pulls images from ECR
│   │       - Sends logs to CloudWatch
│   │       → Like a hotel MANAGER (1 per instance)
│   │
│   ├── Task A (your app container)
│   │   └── Uses: Task Role A (IAM Role) → access S3
│   │
│   └── Task B (another app container)
│       └── Uses: Task Role B (IAM Role) → access DynamoDB
│
└── EC2 Instance #2
    ├── ECS Agent (same Instance Profile)
    └── Task A (same Task Role A)
```

**With Fargate** — no EC2, no Agent, no Instance Profile:
```
ECS Cluster (Fargate)
├── Task A → Task Role A (access S3)
├── Task A → Task Role A (access S3)
└── Task B → Task Role B (access DynamoDB)
(AWS handles everything else)
```

**ECS Agent vs Tasks:**

| | ECS Agent | Tasks |
|---|----------|-------|
| **What it is** | Background program (always running) | Your actual app containers |
| **Purpose** | Manages the EC2 instance for ECS | Runs your business logic |
| **How many** | Always **1 per EC2 instance** | **Many** per EC2 instance |
| **You control it?** | No — comes pre-installed | Yes — you define and start/stop them |
| **IAM Role** | Instance Profile | Task Role |

**Hotel analogy:** ECS Agent = hotel **manager** (one per hotel, handles check-ins, talks to headquarters). Tasks = **guests** (many per hotel, each doing their own thing).

- ECS Agent = **manager** (1 per EC2, handles registration/infra). Tasks = **workers** (your apps)
- Each task can have a **different role** → different permissions per container
- Task Role is defined in the **task definition**

### 4. Load Balancer Integrations

| Load Balancer | Recommendation |
|--------------|----------------|
| **ALB** | Recommended — works with EC2 and Fargate, supports most use cases |
| **NLB** | Only for high throughput/performance or AWS Private Link |
| **CLB** | Not recommended — no advanced features, does NOT work with Fargate |

```
Users → ALB → ECS Tasks (in ECS Cluster)
```

### 5. Data Persistence with EFS

- Mount **Amazon EFS** onto ECS tasks for shared storage
- Works with **both** EC2 and Fargate launch types
- Tasks in **any AZ** can share the same data via EFS
- **Best combo:** Fargate + EFS = fully serverless (compute + storage)

```
ECS Cluster
  ├── Task (AZ-a) ──┐
  ├── Task (AZ-b) ──┤── Amazon EFS (shared data)
  └── Task (AZ-c) ──┘
```

- Use case: persistent multi-AZ shared storage for containers

---

## Exam Tips
- **EC2 Launch Type** = you manage EC2 instances, ECS Agent required
- **Fargate** = serverless, no EC2 to manage, exam prefers this
- **EC2 Instance Profile** = for the ECS Agent (EC2 launch type only)
- **ECS Task Role** = for the container's permissions (both launch types), defined in task definition
- **ALB** = best choice for ECS load balancing, works with Fargate
- **CLB** does NOT work with Fargate
- **EFS** = shared persistent storage for ECS tasks across AZs
- "Serverless containers + serverless storage" = Fargate + EFS
