# 9.6 - Route 53 - TTL (Time To Live)

## Key Concepts

### 1. What is TTL?
- **Time To Live** — tells the client how long to **cache** a DNS record before querying again
- Measured in **seconds**
- Mandatory for every DNS record **except Alias records**

```
Client ──► "myapp.example.com?" ──► Route 53
Client ◄── "IP: 54.22.33.44, TTL: 300" ◄── Route 53

For the next 300 seconds:
Client already knows the IP → no DNS query needed → goes directly to web server
```

### 2. High TTL vs Low TTL

| | High TTL (e.g., 24 hours) | Low TTL (e.g., 60 seconds) |
|---|---|---|
| **DNS traffic** | Less traffic → cheaper | More traffic → more expensive |
| **Record changes** | Clients may have outdated records for up to 24h | Changes propagate quickly |
| **Best for** | Stable records that rarely change | Records you plan to change soon |

### 3. Strategy for Changing Records
1. **Lower the TTL** first (e.g., from 24h to 60s)
2. **Wait** for the old TTL to expire (so all clients pick up the new short TTL)
3. **Change the record value**
4. Changes propagate quickly (within 60s)
5. **Raise the TTL** back to a high value

---

## Exam Tips
- TTL = how long the client caches a DNS result before querying again
- High TTL = cheaper but slow to update | Low TTL = more expensive but fast to update
- TTL is mandatory for all records **except Alias**
- To safely change a record: lower TTL first → wait → change record → raise TTL back
