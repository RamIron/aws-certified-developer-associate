# 9.14 - Routing Policy - Geolocation

## Key Concepts

### 1. How Geolocation Routing Works

- Routes based on **where the user is physically located** (NOT latency)
- Location granularity: **Continent → Country → US State** (most precise wins)
- You **must** create a **default record** for users that don't match any location
- Supports health checks

```
User in Germany ──► Route 53 ──► German app (IP 1)
User in France  ──► Route 53 ──► French app (IP 2)
User in Brazil  ──► Route 53 ──► Default / English app (IP 3)
```

### 2. Use Cases

- **Website localization** — serve language-specific versions (German users → German site)
- **Content restriction** — only allow access from certain countries
- **Load balancing** by region

### 3. Geolocation vs Latency

| | Geolocation | Latency |
|--|------------|---------|
| **Based on** | User's physical location | Network speed to AWS region |
| **Purpose** | Content localization / restriction | Best performance |
| **Default record** | Required (for unmatched locations) | Not applicable |
| **Example** | Germany → German site | Germany → closest fast region |

---

## Exam Tips
- Geolocation ≠ Latency — geolocation is about **where the user is**, not speed
- Most precise location wins: US State > Country > Continent > Default
- **Default record is a must** — unmatched users get no response without it
- Use cases: localization, content restriction, regional load balancing
- Mexico is NOT the United States — if you only set "United States", Mexican users hit the default
