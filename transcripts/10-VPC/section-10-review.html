1 — VPC & Subnets Overview
What Is a VPC?

A VPC (Virtual Private Cloud) is your private network inside AWS. You define the IP range, create subnets, and control routing and access.

VPC — Regional Resource
  • Spans ALL Availability Zones in a region
  • You define a CIDR block (e.g., 10.0.0.0/16)
  • Max 5 VPCs per region (soft limit)
  • Each AWS account gets a default VPC
Subnet — AZ-Level Resource
  • Lives inside one specific AZ
  • Sub-range of the VPC CIDR (e.g., 10.0.1.0/24)
  • AWS reserves 5 IPs per subnet (first 4 + last 1)
  • Public or Private depending on route table
Public Subnet
Has a route to an Internet Gateway in its route table. Resources can be accessed from the internet (if allowed by SG). Use for: ELB, Bastion hosts, NAT Gateway.
Private Subnet
No route to an Internet Gateway. Isolated from the public internet. Use for: application servers, databases, caches. Can reach internet outbound only via NAT.
Default VPC: Every AWS account has one. All subnets are public (one per AZ). CIDR block is 172.31.0.0/16. Instances launched here get both a public and private IP by default. Safe to use for learning — do NOT delete it.

3 — NACL vs Security Groups
Two Layers of Traffic Filtering

AWS provides two distinct firewall mechanisms. Understanding the difference is critical for the exam.

Attribute NACL (Network ACL) Security Group
Level Subnet boundary Instance / ENI boundary
Rule types Allow Deny Allow only
Stateful? Stateless — must allow return traffic Stateful — return traffic auto-allowed
Rule references IP addresses / CIDR only IPs, CIDRs, or other Security Groups
Rule evaluation Rules evaluated in number order (lowest first) All rules evaluated together
Default behavior Default NACL allows everything. Custom NACL denies everything. Denies all inbound, allows all outbound
4 — VPC Flow Logs
Capturing IP Traffic for Visibility & Troubleshooting

VPC Flow Logs capture metadata about all IP traffic going to and from network interfaces. They are the primary tool for diagnosing connectivity issues.

VPC
VPC Level
Captures all traffic for every ENI in the entire VPC. Broadest scope.
SUB
Subnet Level
Captures all traffic for ENIs within a specific subnet.
ENI
ENI Level
Captures traffic for a single Elastic Network Interface. Most granular.
What Gets Captured
  • ALLOWED and DENIED traffic both logged
  • Source IP, destination IP, port, protocol
  • Packet count, byte count, time window
  • ACCEPT or REJECT action recorded
  • AWS managed service traffic: ELB, RDS, ElastiCache, Redshift, NAT Gateway, Transit Gateway
Destinations
  • Amazon S3 — cost-effective long-term storage, queryable with Athena
  • CloudWatch Logs — real-time monitoring, metric filters
  • Kinesis Data Firehose — stream to third-party analytics tools
Exam trap: Flow Logs do NOT capture traffic to/from the AWS DNS server (169.254.169.253), the link-local address (169.254.169.254 for metadata), or DHCP traffic. Use Flow Logs to diagnose NACL vs Security Group issues — look for ACCEPT vs REJECT in the logs.
5 — VPC Peering
Connecting VPCs Privately via the AWS Network

VPC Peering lets two VPCs communicate as if they were on the same private network — no internet involved. It works within the same account, across accounts, and across regions.

Requirements & Constraints
  • IP CIDR ranges must NOT overlap
  • NOT transitive — each pair needs its own peering connection
  • Both VPCs need updated route tables after peering
  • Works across different AWS accounts
  • Works across different regions
Non-Transitivity Explained

If A–B peered and B–C peered, A cannot reach C automatically. You must create A–C peering separately.

VPC-A →← VPC-B Peered
VPC-A →← VPC-C Peered
VPC-B ✕ VPC-C No path!
Key exam point: VPC Peering is NOT transitive. If you need any-to-any connectivity between many VPCs, consider AWS Transit Gateway instead (hub-and-spoke model, supports transitive routing).
6 — VPC Endpoints
Private Access to AWS Services — No Internet Required

VPC Endpoints let resources in your private subnets reach AWS services without traffic ever leaving the AWS network.

Gateway Endpoint
  • Supports S3 and DynamoDB ONLY
  • Implemented as a route in the VPC route table
  • No ENI — no per-hour cost (completely FREE)
  • Associated with specific route tables in specific subnets
  • Does not require a private DNS name
Interface Endpoint (PrivateLink)
  • Supports all other AWS services (SQS, SNS, KMS, etc.)
  • Creates an ENI in your subnet — uses private IP
  • Billed per hour + per GB of data processed
  • Provides private DNS name for the service
  • Can be used from Direct Connect / Site-to-Site VPN
Free tip for the exam: Gateway Endpoint for S3 is free AND the recommended approach for EC2/Lambda in private subnets needing S3 access. Interface Endpoints cost money but support every service.
7 — Site-to-Site VPN vs Direct Connect
Connecting On-Premises Networks to AWS

When your corporate data center needs to communicate with your AWS VPC, you have two main options.

Attribute Site-to-Site VPN Direct Connect (DX)
Path Public internet (encrypted via IPSec) Dedicated private fiber — bypasses internet
Setup time Minutes At least 1 month (physical provisioning)
Bandwidth Limited by internet connection 1 Gbps, 10 Gbps, or 100 Gbps options
Cost Low Higher (port-hour + data transfer)
Use case Quick setup, backup connectivity, low-throughput High-throughput, consistent latency, compliance
AWS-side component for VPN: Virtual Private Gateway (VGW) attached to the VPC, plus a Customer Gateway (CGW) on the on-premises side. Direct Connect uses a Direct Connect Gateway to connect to one or more VPCs.
8 — Three-Tier Architecture
Classic Web Application Layout in a VPC

AWS best practice distributes web applications across three tiers with specific subnet placement for security and scalability.

PUBLIC
Tier 1 — Web / Load Balancer
Elastic Load Balancer (ELB) — publicly accessible, spread across multiple public subnets in different AZs. Receives internet traffic and distributes to application tier. Also: Bastion hosts if needed.
PRIVATE
Tier 2 — Application (EC2 in ASG)
EC2 instances in Auto Scaling Group — private subnets across AZs. Security group only allows inbound from the ELB security group. EFS can mount here for shared file storage (e.g., WordPress images). Outbound via NAT Gateway.
PRIVATE
Tier 3 — Data (RDS + ElastiCache)
RDS (with Multi-AZ standby) and ElastiCache (Redis/Memcached) — deepest private subnets. Security groups only allow inbound from EC2 security groups. No internet access whatsoever.
LAMP Stack / WordPress note: Linux + Apache + MySQL (RDS) + PHP on EC2. WordPress stores media/uploads on Amazon EFS (shared across all EC2 instances in the ASG). ElastiCache (Redis) handles session data and query caching.
9 — Exam Tips & Flash Cards
If the question says "public subnet" →
Route table has a route to an Internet Gateway
Private subnet needs outbound internet access →
NAT Gateway (deployed in PUBLIC subnet, not private)
NAT Gateway vs NAT Instance →
NAT Gateway = AWS-managed, highly available. NAT Instance = self-managed EC2 (legacy, avoid)
Subnet-level firewall, can DENY traffic →
NACL — stateless, evaluates allow AND deny rules in order
Instance-level firewall, return traffic auto-allowed →
Security Group — stateful, allow rules only
Need to troubleshoot "why can't EC2 reach service X?" →
Enable VPC Flow Logs — check ACCEPT vs REJECT records
Connect two VPCs privately, same or cross-account →
VPC Peering — no overlapping CIDRs, NOT transitive
EC2 in private subnet needs to reach S3 or DynamoDB →
Gateway VPC Endpoint (FREE, no internet needed)
Private access to SQS, SNS, KMS, etc. from VPC →
Interface VPC Endpoint (PrivateLink, creates ENI)
On-premises to AWS — need it NOW, low bandwidth ok →
Site-to-Site VPN (minutes to set up, over public internet)
On-premises to AWS — dedicated, consistent, high-speed →
Direct Connect (1+ month to provision, private fiber)
Three-tier: what goes in public vs private subnets? →
PUBLIC: ELB only. PRIVATE: EC2 (app), RDS, ElastiCache (data)
