# 15.7 - CloudFront - ALB/EC2 as an Origin

## Key Concepts

### 1. VPC Origins (New & Recommended Way)

Deliver content from applications in **private subnets** — nothing needs to be public.

```
Users → CloudFront Edge Locations → VPC Origin → Private ALB / NLB / EC2
                                                   (private subnet)
```

- Supports: private **ALB**, **NLB**, and **EC2 instances**
- Most secure setup — applications stay fully private
- CloudFront handles the connection into your VPC

### 2. Public Origins (Old Way)

Before VPC origins, the origin had to be **public**:

**EC2 as origin:**
```
Users → CloudFront Edge → Public EC2 Instance
                          Security Group must allow
                          all CloudFront edge location IPs
```

**ALB as origin:**
```
Users → CloudFront Edge → Public ALB → Private EC2 Instances
                          Security Group must allow
                          all CloudFront edge location IPs
```

- Must find the list of CloudFront public IPs and add them to security groups
- Tedious and risky — if someone modifies the security group, the origin becomes exposed

### 3. VPC Origins vs Public Origins

| | VPC Origins (new) | Public Origins (old) |
|---|-------------------|---------------------|
| **Origin visibility** | Private (in private subnet) | Must be public |
| **Security** | Most secure — fully private | Risk if security group is changed |
| **Setup** | Simple — CloudFront connects via VPC origin | Must manage CloudFront IP list in security groups |
| **Supports** | Private ALB, NLB, EC2 | Public ALB, public EC2 |

---

## Exam Tips
- **VPC origins** = recommended, most secure, resources stay **private**
- Old way = public ALB/EC2 + security group with CloudFront IPs (tedious, risky)
- "Most secure way to connect CloudFront to backend" → **VPC origin**
- VPC origins work with private ALB, NLB, and EC2 instances
