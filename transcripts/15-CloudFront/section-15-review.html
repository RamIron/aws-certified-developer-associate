01 — CloudFront Overview
What is CloudFront?

CloudFront is AWS's CDN (Content Delivery Network). On the exam, CDN = CloudFront. It caches content at ~216 Points of Presence (edge locations) worldwide, serving users from the nearest edge for low latency.

🌍
~216 Edge Locations
Content cached globally. Users served from nearest PoP. Dramatically reduces latency for worldwide audiences.
🛡
DDoS Protection
Built-in via AWS Shield (Standard). Integration with AWS WAF for layer 7 rules and bot protection.
🔒
HTTPS + OAC
HTTPS everywhere. Origin Access Control (OAC) restricts S3 buckets to CloudFront-only access.
CloudFront Origins — What CloudFront Can Serve From
Origin Type Examples Access / Security Notes
S3 Bucket Static files, media, backups OAC + bucket policy Can also use CloudFront for uploads (PUT/POST)
VPC Origin Private ALB, NLB, EC2 in private subnets Most secure (recommended) No public exposure; resources stay private
Custom HTTP S3 website endpoint, public ALB, any HTTP server Direct HTTP Origin must be publicly accessible
CloudFront vs S3 Cross-Region Replication
CloudFront
  • ~216 edge locations worldwide
  • TTL ~1 day (cached content may be slightly stale)
  • Best for static content that needs global reach
  • Files cached at the edge, great cache hit ratio
  • Content available everywhere with no manual setup per region
S3 Cross-Region Replication (CRR)
  • Must be configured for each specific region
  • Near real-time replication — data always fresh
  • Best for dynamic content or low-latency access in a few specific regions
  • Read-only in destination regions
  • No caching — requests go to the replicated bucket
02 — Caching: Cache Key & Policies
How CloudFront Caching Works

CloudFront caches responses at edge locations. The Cache Key determines what counts as a unique cached item. By default it is just the hostname + URL path. You can extend it via Cache Policy.

Default Cache Key: hostname + resource portion of URL (e.g. d123.cloudfront.net/images/cat.jpg). Adding more dimensions (headers, cookies, query strings) creates a more granular cache key but reduces your cache hit ratio.
Cache Policy vs Origin Request Policy
Policy What It Controls Affects Cache Key? Forwarded to Origin?
Cache Policy Cache Key composition + TTL (min, max, default) Yes Yes
Origin Request Policy Additional data sent to origin (headers, cookies, query strings NOT in cache key) No Yes
Cache Behaviors — Route URL Patterns to Different Origins

Cache Behaviors let you route different URL patterns to different origins with different TTLs, headers, and cache settings. The most specific pattern matches first. The default /* is always evaluated last.

Cache Invalidation — Force-Refresh Edge Caches

When you update origin content, edges still serve stale cached versions until TTL expires. Invalidation force-removes files from all edge locations immediately.

  • /* — invalidate all cached content
  • /images/* — invalidate a specific path pattern
  • /index.html — invalidate a single file
Exam trigger: "Updated origin but users still see old content" → answer is Cache Invalidation.

04 — VPC Origins vs Public Origins
Connecting CloudFront to Private Resources

VPC Origins is the modern, recommended approach for connecting CloudFront to ALBs, NLBs, and EC2 instances. It keeps your origin resources private and removes the need for public IPs.

VPC Origins (Recommended) Public Origins (Legacy)
Subnet Private subnets — no public exposure Must be in public subnets with public IPs
Security Most secure — traffic stays within AWS Must whitelist CloudFront IP ranges in Security Groups
IP Management None required — private access Must track and update CloudFront IP list in SGs
Supported ALB, NLB, EC2 instances Public ALB, public EC2, custom HTTP endpoints
Exam signal: "Most secure CloudFront to ALB setup" or "CloudFront without exposing ALB publicly" → answer is VPC Origins.

06 — Advanced Features
Price Classes, Origin Groups, Field-Level Encryption & More

These advanced features each have a specific exam trigger pattern. Know what problem each one solves.

Price Classes — Cost Control

Control which edge locations serve your distribution to manage cost.

  • Class 100 — North America + Europe only. Cheapest option.
  • Class 200 — Most regions except the most expensive.
  • Class All — All edge locations worldwide. Best performance, highest cost.
Exam: "Cheapest CloudFront setup" → Price Class 100
Origin Groups — Failover & DR

Define a primary + secondary origin. CloudFront automatically fails over on HTTP errors (5xx).

  • Provides high availability and failover
  • Combined with S3 CRR = regional disaster recovery
  • Primary fails → CloudFront switches to secondary seamlessly
Exam: "CloudFront regional DR" → Origin Groups + S3 CRR
Field-Level Encryption

Encrypt specific POST fields at the edge using your public key. Only your web server (with the private key) can decrypt.

  • Encrypt up to 10 fields per request
  • CloudFront, ALB, and other layers cannot read the encrypted fields
  • Protects sensitive data (credit cards, SSNs) in transit through the stack
  • Encrypted all the way from edge to application
Exam: "Encrypt sensitive fields, only app server decrypts" → Field-Level Encryption
Real-Time Logs

CloudFront can deliver real-time access logs to a single destination.

  • Only destination: Kinesis Data Streams
  • Configurable sampling rate (% of requests to log)
  • Can filter by specific cache behaviors/paths
  • From Kinesis: forward to Lambda, S3, OpenSearch, etc.
Exam: "CloudFront real-time logs" → Kinesis Data Streams (only option)
Geo Restriction

Block or allow users based on country. Uses a third-party Geo-IP database to determine origin country.

  • Allowlist — Only listed countries can access
  • Blocklist — Listed countries are blocked (403)
  • Country determined by Geo-IP database (not 100% accurate)
  • Use case: copyright law compliance by geography
OAC — Origin Access Control

Restricts S3 bucket access so only CloudFront can fetch objects. Replaces the older OAI (Origin Access Identity).

  • S3 bucket policy grants CloudFront service principal access
  • Bucket can be completely private — no public URLs
  • OAC supports SSE-KMS encrypted S3 objects (OAI did not)
  • OAC is the current recommended approach (not OAI)
Exam: "Secure S3 behind CloudFront" → OAC + S3 bucket policy
