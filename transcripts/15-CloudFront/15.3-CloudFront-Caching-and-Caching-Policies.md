# 15.3 - CloudFront - Caching & Caching Policies

## Key Concepts

### 1. How CloudFront Caching Works

- Cache lives at **each edge location** (as many caches as edge locations)
- Each cached object is identified by a **Cache Key**
- Flow: request → edge checks cache → hit? serve it / miss? fetch from origin → cache result
- Goal: **maximize cache hit ratio** = minimize requests to origin
- Objects expire based on **TTL** — or you can force removal via **invalidation**

### 2. Cache Key (Default)

By default, a Cache Key = **hostname + resource portion of the URL**

```
https://mywebsite.com/content/stories/example-story.html
        ↑ hostname       ↑ resource portion
```

- **Hostname:** the domain name (e.g., `mywebsite.com`)
- **Resource portion:** the path after the domain (e.g., `/content/stories/example-story.html`)
- Any request with the same hostname + resource = **cache hit**

### 3. Enhanced Cache Key

Sometimes content varies by user, device, language, etc. You can add to the Cache Key:
- **HTTP headers** (e.g., language)
- **Cookies** (e.g., session preferences)
- **Query strings** (e.g., `?border=red&size=large`)

More items in Cache Key = more granular caching but **worse cache hit ratio**

**Example:** 100 users request `/blog`
- **None (no headers in key):** Key = `mywebsite.com/blog` → 1 miss, 99 hits → **99% hit ratio**
- **All (all headers in key):** Each user has different headers (browser, language, encoding) → each generates a unique key → almost every request is a **cache miss** → origin gets hammered

### 4. Cache Policy

Defines **what goes into the Cache Key** + **TTL settings**:

| Component | Options |
|-----------|---------|
| **Headers** | None, Whitelist |
| **Cookies** | None, Whitelist, All, All-Except |
| **Query Strings** | None, Whitelist, All-Except, All |
| **TTL** | 0 seconds to 1 year (also controlled via `Cache-Control` or `Expires` headers) |

**Key rule:** Everything included in the Cache Key is **automatically forwarded** to the origin.

- Can create custom policies or use AWS **managed policies**
- None = best caching performance (fewer unique keys)
- All = worst caching performance (many unique keys)

### 5. Origin Request Policy

Forwards extra data to the origin **WITHOUT adding it to the Cache Key**.

- Add extra headers, cookies, query strings for the origin to use
- Can add custom headers (e.g., API key, secret header) not present in the viewer request
- Does **NOT** affect caching — only affects what the origin receives

### 6. Cache Policy vs Origin Request Policy

```
Request (headers, cookies, query strings)
    │
    ├── Cache Policy → defines Cache Key (hostname + resource + selected items)
    │                   these items ARE forwarded to origin
    │
    └── Origin Request Policy → extra items forwarded to origin
                                 but NOT part of Cache Key
```

Example:
- **Cache on:** hostname, resource, `Authorization` header
- **Also forward to origin:** `User-Agent`, `Session-ID`, `?ref=` query string
- Caching only uses the cache policy items — origin gets both

---

## Exam Tips
- Default Cache Key = **hostname + resource URL**
- Cache Policy = what's in the Cache Key + TTL (also forwarded to origin)
- Origin Request Policy = forwarded to origin but **NOT** in Cache Key
- More in Cache Key = more granular but worse hit ratio
- "Origin needs data but you don't want to break caching" → **Origin Request Policy**
- TTL can be set via Cache Policy or `Cache-Control` / `Expires` headers
