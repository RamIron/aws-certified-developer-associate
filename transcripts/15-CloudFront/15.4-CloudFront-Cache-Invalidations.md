# 15.4 - CloudFront - Cache Invalidations

## Key Concepts

### 1. The Problem

- You update files at the origin (e.g., S3 bucket)
- Edge locations still serve the **old cached version** until TTL expires
- You don't want to wait (e.g., TTL = 1 day)

### 2. The Solution: Cache Invalidation

Force CloudFront to **remove cached files** from all edge locations, bypassing the TTL.

- Next request = cache miss → edge fetches the **updated file** from the origin

### 3. Invalidation Paths

| Path | Effect |
|------|--------|
| `/*` | Invalidate **all files** |
| `/images/*` | Invalidate all files under `/images/` |
| `/index.html` | Invalidate a **specific file** |

### 4. How It Works

```
1. You update files in S3 (origin)
2. You create an invalidation: /index.html + /images/*
3. CloudFront removes those files from ALL edge location caches
4. Next user request → cache miss → edge fetches new version from origin
```

---

## Exam Tips
- "Updated origin but users see old content" → **Cache Invalidation**
- Can invalidate specific files or use wildcards (`/*`, `/images/*`)
- Invalidation removes from cache, forcing re-fetch from origin
- Alternative to waiting for TTL to expire
