# 15.9 - CloudFront Signed URL / Cookies

## Key Concepts

### 1. Why Signed URLs / Cookies?

- Make CloudFront distribution **private** (premium/paid content)
- Control **who** can access **what** content
- Know who has access to what

### 2. Signed URL vs Signed Cookie

| | Signed URL | Signed Cookie |
|---|-----------|---------------|
| **Scope** | Access to **one file** per URL | Access to **multiple files** with one cookie |
| **Use when** | You have individual files to share | You have many files users need access to |

### 3. Policy Attached to Signed URL/Cookie

When creating a signed URL or cookie, you define:
- **Expiration** — when it stops working (minutes for movies, years for long-term private content)
- **IP range** — restrict which client IPs can use it
- **Trusted signers** — which AWS accounts can create signed URLs

### 4. How It Works

```
1. Client authenticates with your application
2. Your app uses AWS SDK to generate a CloudFront signed URL
3. Signed URL returned to client
4. Client uses signed URL to access content via CloudFront
```

### 5. CloudFront Signed URL vs S3 Pre-signed URL

| | CloudFront Signed URL | S3 Pre-signed URL |
|---|----------------------|-------------------|
| **Works with** | Any origin (S3, EC2, ALB, HTTP) | S3 only |
| **Access through** | CloudFront distribution | Directly to S3 bucket |
| **Key type** | Account-wide key-pair (root manages) | IAM principal's own credentials |
| **Caching** | Leverages CloudFront edge caching | No caching (direct S3 access) |
| **Use when** | S3 is behind CloudFront with OAC | Users access S3 directly (no CloudFront) |

### 6. When to Use Which

```
S3 behind CloudFront (OAC blocks direct access)?
  → CloudFront Signed URL (can't bypass CloudFront to reach S3)

Users access S3 directly (no CloudFront)?
  → S3 Pre-signed URL
```

### 7. Signers: How Signed URLs Get Created

To generate signed URLs/cookies, CloudFront needs to verify they came from a trusted source. This is done with **RSA key pairs**: your app signs with the private key, CloudFront verifies with the public key. There are two ways to manage these keys:

#### Trusted Key Groups vs CloudFront Key Pairs

| | Trusted Key Group (new, recommended) | CloudFront Key Pair (old) |
|---|--------------------------------------|--------------------------|
| **Who manages** | Any IAM user with permissions | Root account only |
| **Automation** | Yes — APIs to create/rotate keys | No APIs — console only |
| **How it works** | Upload public key to CloudFront, private key on your app (EC2) to sign URLs | Root creates key pair in security credentials |

- You generate an RSA key pair (2,048 bit)
- **Private key** → your application (e.g., EC2) uses it to sign URLs
- **Public key** → uploaded to CloudFront to verify signatures
- Key group can hold up to **5 public keys**

---

## Exam Tips
- **Signed URL** = one file, **Signed Cookie** = multiple files
- CloudFront Signed URL works with **any origin** (not just S3)
- S3 Pre-signed URL = direct S3 access, inherits **signer's IAM permissions**
- "Premium content via CloudFront" → **Signed URL or Signed Cookie**
- "S3 behind CloudFront with OAC, need temporary access" → **CloudFront Signed URL** (not S3 pre-signed)
- "Direct S3 access without CloudFront" → **S3 Pre-signed URL**
