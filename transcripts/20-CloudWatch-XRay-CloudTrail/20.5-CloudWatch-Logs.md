# 20.5 - CloudWatch Logs

> Udemy Lecture #239 · Section 20: AWS Monitoring & Audit: CloudWatch, X-Ray and CloudTrail

## Key Concepts

### 1. Log Structure: Groups → Streams

```
CloudWatch Logs
├── Log Group (= your application)
│   ├── Log Stream (= instance / container / log file)
│   ├── Log Stream
│   └── Log Stream
├── Log Group
│   └── ...
```

- **Log Group**: represents an application (name it whatever you want)
- **Log Stream**: represents a specific instance, container, or log file within that group
- **Expiration policy**: never expire (indefinite), or from **1 day to 10 years**
- All logs **encrypted by default** — optionally use your own **KMS keys**

### 2. What Sends Logs to CloudWatch?

| Source | What it sends |
|--------|--------------|
| SDK / CloudWatch Unified Agent | Custom application logs |
| CloudWatch Logs Agent | Legacy (deprecated, replaced by Unified Agent) |
| Elastic Beanstalk | Application logs |
| ECS | Container logs |
| Lambda | Function logs |
| VPC Flow Logs | Network traffic metadata |
| API Gateway | All API requests |
| CloudTrail | API calls (filtered) |
| Route 53 | DNS queries |

### 3. CloudWatch Logs Insights (Query Engine)

- Write queries to search and analyze log data
- Specify a timeframe, get visualizations + raw log lines
- Auto-detects fields from your logs
- Can filter, aggregate, sort, limit
- Export results or add to dashboards
- Can query **multiple log groups** across **different accounts**
- **NOT real-time** — queries historical data only

### 4. Exporting Logs: Batch vs Real-Time

| Method | Destination | Speed | API / Feature |
|--------|------------|-------|---------------|
| **Batch export** | S3 | Up to **12 hours** | `CreateExportTask` API |
| **Subscription filter** | KDS, KDF, Lambda | **Real-time** stream | Subscription filter |

Subscription filter destinations:
- **Kinesis Data Streams** → then to KDF, Kinesis Analytics, EC2, Lambda
- **Kinesis Data Firehose** → near real-time to S3, OpenSearch
- **Lambda** → custom processing, or managed function to OpenSearch

### 5. Cross-Account Log Aggregation

Multiple accounts/regions can send logs to a single destination using subscription filters:

```
Account A (sender)                    Account B (recipient)
┌─────────────────┐                  ┌─────────────────────┐
│ CW Logs         │                  │ Kinesis Data Stream  │
│   ↓             │                  │   ↓                  │
│ Subscription    │──── destination ──→│ KDF → S3            │
│ Filter          │    access policy  │                     │
└─────────────────┘                  └─────────────────────┘
```

Requirements (two policies needed):

1. **IAM Role (Cross-Account)** in recipient account:
   - Action: `kinesis:PutRecord`
   - Resource: the KDS ARN in recipient account
   - Must be **assumable** by sender account

2. **Destination Access Policy** on the subscription destination:
   - Principal: sender account ID (e.g., `111111111111`)
   - Action: `logs:PutSubscriptionFilter`
   - Resource: the destination ARN in recipient account

---

## Exam Tips

- Log Group = app, Log Stream = instance/container
- Retention: **1 day to 10 years** (or never expire)
- Batch export to S3: up to **12 hours** → `CreateExportTask`
- Real-time export: **Subscription filters** → KDS, KDF, Lambda
- Logs Insights = **query engine, NOT real-time**
- Cross-account aggregation uses subscription destinations + access policies
