# 20.14 - Amazon EventBridge

## Overview
Amazon EventBridge (formerly CloudWatch Events) is a serverless event bus that lets you react to events from AWS services, SaaS partners, and your own applications.

## Key Concepts

### Two Ways to Use EventBridge
1. **Schedule (Cron)** — "Every hour, trigger a Lambda function"
2. **Event Pattern** — React when something happens (e.g., IAM root user sign-in → SNS email)

### Event Sources → Filter → JSON Event → Destinations

```
SOURCES                          DESTINATIONS
┌──────────────┐                 ┌──────────────────┐
│ EC2 (start/  │                 │ Lambda            │
│  stop/term)  │                 │ AWS Batch         │
│ CodeBuild    │──┐              │ ECS Task          │
│ S3 events    │  │   ┌──────┐  │ SQS / SNS / KDS   │
│ Trusted Adv. │  ├──→│ Event│  │ Step Functions     │
│ CloudTrail   │  │   │Bridge├─→│ CodePipeline       │
│  (any API)   │──┘   │      │  │ CodeBuild          │
│ Schedule/Cron│──────→│Filter│  │ SSM Automation     │
└──────────────┘      └──────┘  │ EC2 Actions        │
                                └──────────────────┘
```

- Events are **JSON documents** with details (instance ID, time, IP, etc.)
- Filters let you narrow events (e.g., "only events for bucket X")
- **CloudTrail + EventBridge** = intercept ANY API call in your account

### Three Types of Event Buses

| Bus Type | Source | Example |
|----------|--------|---------|
| **Default Event Bus** | AWS services | EC2, S3, CodeBuild, CloudTrail |
| **Partner Event Bus** | SaaS partners | Zendesk, Datadog, Auth0 |
| **Custom Event Bus** | Your own apps | Your microservices sending custom events |

### Event Archiving & Replay
- Archive **all events** or a **filtered subset**
- Retention: **indefinite** or a **set period**
- Can **replay archived events** — great for debugging (fix a Lambda bug, then replay the event that caused it)

### Schema Registry
- EventBridge **analyzes events** in your bus and **infers the schema**
- You can **download generated code** that already knows the event structure
- Schemas can be **versioned** over time

### Resource-Based Policies
- Manage permissions for a specific event bus
- Allow/deny events from **other accounts or regions**
- Use case: **central event bus** in an AWS Organization that aggregates events from multiple accounts via `PutEvents`

## Exam Tips
- EventBridge = formerly **CloudWatch Events** (same service, new name)
- Cron jobs in AWS → **EventBridge schedule**
- React to any AWS API call → **CloudTrail + EventBridge**
- Three bus types: Default, Partner, Custom
- Replay archived events → **debugging/troubleshooting**
- Cross-account event bus → **resource-based policies**
- Schema Registry → **auto-generate code** from event structure

## Cross-Account Resource-Based Policy Example
```json
{
  "Effect": "Allow",
  "Action": "events:PutEvents",
  "Principal": { "AWS": "111122223333" },
  "Resource": "arn:aws:events:us-east-1:123456789012:event-bus/central-event-bus"
}
```
Account 111122223333 sends events → central-event-bus in account 123456789012
