# 20.16 - Amazon EventBridge - Multi-Account Aggregation

## Overview
Short architecture lesson on how to aggregate events from multiple AWS accounts into a single central event bus.

## Architecture Pattern

```
Account A                    Central Account
┌─────────────────┐         ┌──────────────────────────┐
│ EC2 state change │         │                          │
│       │          │         │  Event Bus               │
│       ▼          │         │  (resource-based policy   │
│  Event Rule ─────┼────────→│   allows A, B, C, D)     │
└─────────────────┘         │       │                   │
                            │       ▼                   │
Account B                   │  Event Rule               │
┌─────────────────┐         │       │                   │
│ EC2 state change │         │       ▼                   │
│       │          │         │  SNS / Lambda / etc.      │
│       ▼          │         └──────────────────────────┘
│  Event Rule ─────┼────────→
└─────────────────┘

(Same for Account C, D, etc.)
```

## How It Works
1. Each account defines an **event rule** that matches the events you want to forward (e.g., EC2 state changes)
2. The **target** of that rule is the **event bus in the central account**
3. The central account's event bus needs a **resource-based policy** allowing `events:PutEvents` from the other accounts
4. The central account then creates its **own rules** on the event bus to trigger actions (SNS, Lambda, etc.)

## Key Takeaway
- An event rule's target can be an **event bus in another account**
- The receiving bus must have a **resource-based policy** granting access
- This is the standard pattern for centralized event monitoring across an AWS Organization

- Multi-account event aggregation → **EventBridge + resource-based policy on central bus**
