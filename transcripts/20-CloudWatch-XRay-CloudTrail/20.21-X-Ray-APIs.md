# 20.21 - X-Ray APIs

## Overview
The X-Ray APIs you need to know for the exam, split into Write APIs (used by the daemon) and Read APIs (used by the console/your code to retrieve data).

## Key Concepts

### Write APIs (used by X-Ray Daemon)

Managed policy: `AWSXRayWriteOnlyAccess`

| API | Purpose |
|-----|---------|
| **PutTraceSegments** | Upload segment documents to X-Ray |
| **PutTelemetryRecords** | Upload daemon metrics (segments received, rejected, connection errors) |
| **GetSamplingRules** | Retrieve current sampling rules (so daemon knows what to sample) |
| **GetSamplingTargets** | Advanced — related to sampling rules |
| **GetSamplingStatisticsSummaries** | Advanced — related to sampling rules |

> **Why is there a "Get" in the Write policy?** Because the daemon needs to **read** the sampling rules to know which requests to send to X-Ray. Without this, changing sampling rules in the console wouldn't propagate to the daemon.

### Read APIs (used by console / your applications)

| API | Purpose |
|-----|---------|
| **GetServiceGraph** | Get the main visual service map |
| **BatchGetTraces** | Retrieve a list of traces by their IDs (each trace = collection of segments from one request) |
| **GetTraceSummaries** | Get trace IDs and annotations for a time range |
| **GetTraceGraph** | Get the service graph for specific trace IDs |

### Read API Workflow
```
GetTraceSummaries (get IDs for a time range)
        │
        ▼
BatchGetTraces (get full traces by ID)
        │
        ▼
GetTraceGraph (get service graph for those traces)
```

## Exam Tips
- **Write policy** has 2 Puts + 3 Gets (the Gets are for sampling rules)
- **Read policy** is all Gets + 1 BatchGet
- `PutTraceSegments` = the core API to send trace data to X-Ray
- `GetSamplingRules` is in the WRITE policy because the daemon needs it to know what to sample
- If X-Ray daemon can't write → check IAM policy has `AWSXRayWriteOnlyAccess`
