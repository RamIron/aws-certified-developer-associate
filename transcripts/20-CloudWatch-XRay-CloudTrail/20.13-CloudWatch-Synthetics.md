# 20.13 - CloudWatch Synthetics Canary

## Overview
Configurable scripts that programmatically reproduce what customers do on your site/API, catching issues before real users hit them.

## Why It Exists
You want to know your checkout flow, login page, or API is broken **before** your customers discover it — not after.

## Key Concepts

### What It Does
- Runs scripts that simulate user behavior (click buttons, fill forms, navigate pages)
- Monitors APIs, URLs, and websites
- Checks availability and latency of endpoints
- Stores load time data and **takes screenshots of the UI**

### Script Runtime
- Written in **Node.js** or **Python**
- Has access to a **headless Google Chrome browser** (can do anything Chrome can do)
- Can run **once** or on a **regular schedule**

### Architecture Example
```
CloudWatch Synthetics Canary
    monitors app in us-east-1
            │
            ▼ (failure detected)
    CloudWatch Alarm
            │
            ▼
    Lambda Function
            │
            ▼
    Update Route 53 DNS → redirect to us-west-2 (healthy app)
```

### Blueprints (6 types)

| Blueprint | Purpose |
|-----------|---------|
| **Heartbeat Monitor** | Load URL, store screenshots & HTTP archive, verify it works |
| **API Canary** | Test read/write functions of REST APIs |
| **Broken Link Checker** | Check all links inside a URL for broken links |
| **Visual Monitoring** | Compare screenshot vs baseline screenshot |
| **Canary Recorder** | Record actions on a website → auto-generate script |
| **GUI Workflow Builder** | Verify web actions (e.g., login form) work correctly |

## Exam Tips
- Synthetics Canary = **proactive monitoring** (find issues before customers do)
- Scripts run on **Node.js or Python** with **headless Chrome**
- Know the blueprint names and what each does
- Can trigger CloudWatch Alarms → Lambda → automated remediation
