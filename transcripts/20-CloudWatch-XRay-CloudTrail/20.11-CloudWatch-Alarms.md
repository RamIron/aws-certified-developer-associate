# 20.11 - CloudWatch Alarms

> Udemy Lecture #245 · Section 20: AWS Monitoring & Audit: CloudWatch, X-Ray and CloudTrail

## Key Concepts

### 1. Alarm Basics

Alarms trigger notifications based on metrics. Three states:

| State | Meaning |
|-------|---------|
| **OK** | Metric is within threshold — all good |
| **INSUFFICIENT_DATA** | Not enough data to determine state |
| **ALARM** | Threshold breached — notification sent |

- **Period**: how long the alarm evaluates the metric (can be 10s, 30s, or multiples of 60s for high-res metrics)

### 2. Alarm Targets (Three Main Actions)

| Target | What it does |
|--------|-------------|
| **EC2 Actions** | Stop, terminate, reboot, or **recover** an instance |
| **Auto Scaling** | Trigger scale out / scale in |
| **SNS** | Send notification → can hook to Lambda for any custom action |

### 3. Composite Alarms

Single alarms monitor **one metric**. Composite alarms combine **multiple alarms** using **AND / OR** conditions.

```
Alarm A (CPU > 80%) ──┐
                       ├── AND ──→ Composite Alarm → SNS
Alarm B (IOPS > 5000) ─┘
```

- Reduces alarm noise — e.g., "only alert me when CPU is high AND network is low"
- Each underlying alarm monitors its own metric
- The composite alarm monitors the **states** of other alarms, not metrics directly

### 4. EC2 Instance Recovery

CloudWatch can monitor three EC2 status checks:

| Check | What it monitors |
|-------|-----------------|
| **Instance status** | The EC2 VM itself |
| **System status** | The underlying hardware |
| **EBS status** | Attached EBS volumes health |

When alarm triggers → **EC2 instance recovery** (moves instance to another host). The recovered instance keeps:
- Same private, public, and Elastic IP
- Same metadata
- Same placement group

You can also send an SNS alert about the recovery.

### 5. Testing Alarms & Metric Filter Integration

- Alarms can be created on top of **CloudWatch Logs Metric Filters** (the full chain: Logs → Metric Filter → Metric → Alarm → SNS)
- To **test an alarm** without waiting for a threshold breach: use the **`set-alarm-state`** CLI call
  - Forces the alarm into ALARM state
  - Useful to verify that the alarm triggers the correct action

---

## Exam Tips

- Three states: **OK**, **INSUFFICIENT_DATA**, **ALARM**
- Three targets: **EC2 actions**, **ASG scaling**, **SNS**
- **Composite Alarms** = AND/OR on multiple alarms → reduces noise
- EC2 recovery keeps same IPs, metadata, placement group
- Test alarms with **`set-alarm-state`** CLI
- Metric filter → alarm chain: Logs → Filter → Metric → Alarm → SNS

## Tricky Exam Questions

**Q: ASG min=2, alarm triggers scale-in when CPU < 60%, already at 2 instances with low traffic. What happens?**
- A: Alarm stays in **ALARM state** but **never decreases** instances below ASG minimum capacity
- Key: ASG minimum capacity **always wins** over CloudWatch Alarm actions. The alarm can't force ASG below its floor.
- The alarm won't go to OK either — CPU is still below 60%. It stays stuck in ALARM doing nothing.

#### 🔥 Tricky: High-Resolution Alarm Period
- Q: A CloudWatch Alarm set on a High-Resolution Custom Metric can be triggered as often as...?
- **Answer: 10 seconds**
- High-resolution custom metrics (StorageResolution = 1) allow alarm periods of **10s or 30s**
- Standard metrics only allow periods in **multiples of 60 seconds**
- Trap: "1 second" sounds right because high-res metrics publish every 1s, but the minimum *alarm* period is 10s
