# 20.17 - X-Ray Overview

## Overview
AWS X-Ray provides visual analysis of distributed applications by tracing requests as they flow through your architecture. It solves the problem of debugging microservices where logs are scattered across hundreds of services.

## Why It Exists
- Debugging distributed systems is a nightmare — logs in different formats, different services, no common view
- Monolith = manageable. 100 microservices all talking to each other = impossible to debug with just CloudWatch Logs
- X-Ray gives you a **visual service map** showing exactly where errors and bottlenecks are

## Key Concepts

### What X-Ray Provides
- **Visual service map** — see all services and how they connect
- **Troubleshoot performance** — identify bottlenecks
- **Understand dependencies** — which microservice calls which
- **Pinpoint issues** — visually see which service is failing (orange/yellow = errors)
- **Per-request behavior** — trace individual requests end-to-end
- **SLA monitoring** — are we meeting latency requirements?
- **User impact** — which users are affected by errors

### Tracing Terminology
| Term | Meaning |
|------|---------|
| **Trace** | End-to-end path of a single request through all services |
| **Segment** | Data from one service/component handling the request |
| **Sub-segment** | More granular detail within a segment |
| **Annotations** | Extra key-value info added to traces for searching/filtering |

### How to Enable X-Ray (TWO things required)

**1. Modify your code — import the X-Ray SDK**
- Supported: Java, Python, Go, Node.js, .NET
- SDK captures: AWS service calls, HTTP/HTTPS requests, database calls (MySQL, PostgreSQL, DynamoDB), queue calls
- Minimal code changes, but still required

**2. Run the X-Ray Daemon (or enable AWS integration)**
- Daemon = low-level **UDP packet interceptor**
- Runs on Linux, Windows, Mac
- Sends traces in **batch every 1 second** to AWS X-Ray
- On EC2/on-premises: you must **install the daemon yourself**
- On Lambda/managed services: daemon runs **automatically**

```
EC2 Instance
┌─────────────────────────────────┐
│  Your Code (with X-Ray SDK)     │
│         │                       │
│         ▼ (traces via UDP)      │
│  X-Ray Daemon                   │
│         │                       │
│         ▼ (batch every 1s)      │
│    AWS X-Ray Service            │
└─────────────────────────────────┘
         │
         ▼
   Service Map (visual graph)
   computed from all segments/traces
```

### Security
- **IAM authorization** for write access to X-Ray
- **KMS** for encryption at rest

### Compatibility
- Lambda, Elastic Beanstalk, ECS, ELB, API Gateway, EC2, on-premises servers

### Troubleshooting X-Ray Not Working

**On EC2:**
1. Check IAM role has proper X-Ray permissions
2. Check X-Ray Daemon is running on the instance

**On Lambda:**
1. Check IAM execution role has X-Ray policy (`AWSXRayWriteOnlyAccess`)
2. Check X-Ray SDK is imported in code
3. Check **Active Tracing** is enabled in Lambda configuration

## Exam Tips
- "X-Ray works locally but not on EC2" → **daemon not running** or **missing IAM role**
- Two things needed: **X-Ray SDK in code** + **X-Ray Daemon running**
- X-Ray collects all segments/traces → computes the **service map automatically**
- You can trace **all requests** or just a **sample** (percentage or rate)
- Non-technical people can use X-Ray (it's visual/graphical)

## Tricky Exam Questions

**Q: X-Ray works on laptop, fails on EC2 deployed via CodeDeploy. Daemon is confirmed running. Why?**
- A: IAM role on EC2 doesn't have X-Ray write permissions
- Trap: "enable inbound UDP traffic" sounds right but UDP is **local** (app → daemon on same instance) — no security group rules needed for localhost traffic

**Q: How to configure X-Ray Daemon to send traces from multiple accounts to a central account?**
- A: Create an IAM role in the central account, then create IAM roles in the other accounts to **assume** this central role
- Trap: "Create an IAM user + access keys" — never use access keys for cross-account. Always use **role assumption** (best practice)

**Q: "By default, the X-Ray SDK records the first request _____, and _____ of any additional requests."**
- A: **each second**, **5%**
- Trap: "each minute" — don't confuse with CloudWatch metric intervals (5 min / 1 min). X-Ray sampling is per **second**
