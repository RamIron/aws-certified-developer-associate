# 20.10 - CloudWatch Logs - Metric Filters Hands On

> Udemy Lecture #244 · Section 20: AWS Monitoring & Audit: CloudWatch, X-Ray and CloudTrail

## Hands-On Summary

### Creating a Metric Filter

1. Go to CloudWatch Logs → select log group → **Metric Filters** tab → Create
2. Enter a **filter pattern** (e.g., `400` to catch HTTP 400 error codes)
3. Test the pattern against sample logs (found 14/50 matches)
4. Configure the metric:
   - Filter name: `MetricFilter400Code`
   - Namespace: `MetricFilters` (custom)
   - Metric name: `MyDemoFilter`
   - Metric value: `1` (publish 1 each time a match occurs)
   - Default value: `0` (when no match)

### Confirming: NOT Retroactive

- After creating the filter, the metric graph shows **zero** — no backfill of old data
- Had to restart Beanstalk app servers and trigger new requests to generate new logs
- Only then did the custom namespace `MetricFilters` appear in CloudWatch Metrics

### Creating an Alarm on the Metric Filter

1. From the metric → **Create Alarm**
2. Set condition: static threshold, greater than 50 (e.g., 50+ HTTP 400 errors)
3. Action: send notification to **SNS topic**
4. Name: `DemoMetricFilterAlarm`

### Full Chain Demonstrated

```
Beanstalk → Nginx logs → CloudWatch Logs → Metric Filter (pattern: "400")
  → CloudWatch Metric → CloudWatch Alarm → SNS notification
```

---

## Key Takeaway

The hands-on confirms the full **Logs → Metric Filter → Metric → Alarm → SNS** chain from the previous lesson, and proves that metric filters are **not retroactive**.
