# 20.28 - CloudTrail vs CloudWatch vs X-Ray

## Overview
One-slide summary of the three monitoring/audit services and when to use each.

## The Big Picture

| Service | Purpose | One-liner |
|---------|---------|-----------|
| **CloudTrail** | **Audit** | Who did what and when? (API calls) |
| **CloudWatch** | **Monitoring** | How are my resources/apps performing? (metrics, logs, alarms) |
| **X-Ray** | **Tracing** | Where is the bottleneck in my distributed app? (traces, service map) |

## Details

### CloudTrail = Audit
- Records API calls made by users, services, and the console
- Detect unauthorized calls
- Find root cause of changes ("who deleted this?")

### CloudWatch = Monitoring
- **Metrics** — resource performance (CPU, memory, custom metrics)
- **Logs** — application log storage and analysis
- **Alarms** — notifications when metrics go wrong

### X-Ray = Tracing
- Automated **trace analysis** for distributed services
- Central **service map visualization**
- Debug **latency, errors, and faults**
- Track requests across distributed systems

## Exam Decision Tree

```
"Who made this API call?"           → CloudTrail
"Is my CPU too high?"               → CloudWatch Metrics
"Why is my app logging errors?"     → CloudWatch Logs
"Alert me when something is wrong"  → CloudWatch Alarms
"Where is the bottleneck?"          → X-Ray
"Which microservice is failing?"    → X-Ray
```

## Exam Tips
- CloudTrail = **API calls** (audit)
- CloudWatch = **metrics + logs + alarms** (monitoring)
- X-Ray = **traces + service map** (debugging distributed systems)
