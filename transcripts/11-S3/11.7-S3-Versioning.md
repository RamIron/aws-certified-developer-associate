# 11.7 - S3 Versioning

## Key Concepts

### 1. How Versioning Works

- Enabled at the **bucket level** (not per object)
- Every time you upload an object with the same key, S3 creates a new **version** (v1, v2, v3...)
- Previous versions are kept — nothing is overwritten

```
Upload coffee.jpg → version 1
Upload coffee.jpg → version 2 (v1 still exists)
Upload coffee.jpg → version 3 (v1 and v2 still exist)
```

### 2. Why Version? (Best Practice)

- **Protects against unintended deletes** — deleting adds a "delete marker", previous versions remain
- **Easy rollback** — restore any previous version at any time

### 3. Important Notes

- Files uploaded **before enabling versioning** have version = **null**
- **Suspending versioning** does NOT delete existing versions (safe operation)
- Deleting a versioned file = adds a **delete marker** (not a real delete)
- To truly delete: you must delete the specific version

---

## Exam Tips
- Versioning is at the **bucket level**
- Pre-versioning files have version **null**
- Delete = delete marker (recoverable), not permanent
- Delete a **specific version** = permanent delete (irreversible)
- Suspending versioning ≠ deleting versions

### Delete Behavior (Important)

```
Delete an object (no version specified):
  → Adds a DELETE MARKER (recoverable, all versions still exist)
  → Remove the delete marker to restore the file

Delete a SPECIFIC VERSION (by version ID):
  → PERMANENT DELETE of that version (gone forever, cannot be undone)
```
