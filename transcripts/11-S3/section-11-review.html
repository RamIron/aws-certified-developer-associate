1 — S3 Basics
Buckets, Objects & Keys

Amazon S3 is object storage that scales infinitely. It is one of AWS's core building blocks — used for backups, static hosting, data lakes, and application storage.

Buckets
  • Top-level container for objects (like a "directory")
  • Name must be globally unique across ALL accounts and ALL regions
  • Bucket itself is a region-level resource (data lives in one region)
  • Naming: 3–63 chars, lowercase letters, numbers, hyphens — no uppercase, no underscores
  • Max 100 buckets per account (soft limit)
Objects & Keys
  • Files stored in buckets are called objects
  • Object key = full path e.g., photos/vacation/beach.jpg
  • Key = prefix (photos/vacation/) + object name (beach.jpg)
  • S3 has NO real directories — the / is just part of the key name
  • Max object size: 5 TB
  • Multi-part upload required for files > 5 GB, recommended > 100 MB
Common confusion on the exam: S3 bucket names are globally unique (across all AWS accounts), but buckets are region-level resources. The S3 console shows all buckets globally, which tricks people. Your data actually lives in the region you selected when creating the bucket.
2 — S3 Security & Access Control
Two Axes of Access Control

Access to S3 objects is granted when an IAM permission OR a resource-based policy allows it, AND there is no explicit deny.

Type Mechanism Applies to
User-Based IAM Policies on users, groups, or roles Specific IAM principals in your account
Resource-Based Bucket Policy (JSON) attached to the bucket Any principal including cross-account and anonymous
IAM
IAM User access
Attach an IAM policy to the user/group granting s3:GetObject etc. Works within the same account. No bucket policy needed.
EC2
EC2 instance access
Attach an IAM Role to the EC2 instance. NEVER store AWS credentials on the instance. The SDK/CLI on EC2 automatically uses the role via the metadata service.
PUB
Public internet access
Bucket Policy with Principal: "*". Block Public Access setting must be disabled first. Use for static website hosting or public file distribution.
XAC
Cross-account access
Bucket Policy specifying the external account's IAM principal. No IAM policy change needed on your side once bucket policy is set.
Block Public Access is an extra safety layer. It overrides bucket policies and ACLs — even if your bucket policy says public, Block Public Access will still block it. Enable at the account level to prevent any bucket in the account from becoming public.
3 — S3 Static Website Hosting
Hosting HTML/CSS/JS Files Directly from S3
How to Enable
  • Enable "Static website hosting" in bucket Properties
  • Specify index document (e.g., index.html)
  • Optionally specify error document (e.g., error.html)
  • S3 generates a website endpoint URL
  • Bucket policy must allow public s3:GetObject
  • Block Public Access must be DISABLED
Key Facts for the Exam
  • 403 Forbidden = bucket policy is not allowing public reads
  • S3 website URL format: bucket.s3-website.region.amazonaws.com
  • Valid as a Route 53 Alias target (no additional cost)
  • Does NOT support HTTPS natively — use CloudFront in front for HTTPS
  • Only supports GET and HEAD requests
Route 53 + S3 website: Create an Alias record pointing to the S3 website endpoint. The bucket name MUST match the domain name exactly (e.g., bucket www.example.com for domain www.example.com).
4 — S3 Versioning
Protecting Objects from Accidental Deletion & Overwrites

When versioning is enabled, every upload to the same key creates a new version instead of overwriting. Each version gets a unique Version ID.

Versioning Behavior
  • Enabled at the bucket level
  • Once enabled, cannot be fully disabled — only suspended
  • Suspending does NOT delete existing versions
  • Files uploaded before enabling versioning have version = null
  • Protects against accidental overwrites and deletes
Delete Behavior
  • Delete without version ID → adds a delete marker (object appears gone but is recoverable — delete the marker)
  • Delete with specific version ID → permanent, irreversible deletion of that version
  • Restoring: delete the delete marker to un-delete an object
Action What happens Recoverable?
Delete (no version specified) Delete marker added, object hidden Yes — delete the marker
Delete by specific Version ID That exact version permanently removed No — permanent
Upload same key again New version created, old version preserved Old version still accessible
Cost implication: Every version is stored and billed. Use Lifecycle Policies to automatically expire old versions (e.g., delete versions older than 90 days) to avoid unbounded storage costs.

Alias record pointing to S3 website endpoint. Bucket name must match domain.
Delete S3 object (versioned bucket) — recoverable? →
Yes — delete marker added. Delete the marker to restore. Delete by version ID = permanent.
S3 Replication requirement →
Versioning MUST be enabled on BOTH source and destination buckets
Existing objects not replicated after enabling replication →
Use S3 Batch Replication to replicate pre-existing objects on demand
Permanent deletes in replication — replicated to target? →
NEVER — preventing malicious deletes from propagating is a security feature
Unknown access pattern, don't want retrieval fees →
S3 Intelligent-Tiering — auto-moves objects, no retrieval charges, small monitoring fee
Glacier, need data in milliseconds →
Glacier Instant Retrieval (milliseconds, min 90 days storage)
Glacier, need data in minutes →
Glacier Flexible — Expedited tier (1–5 min). Standard = 3–5 hrs. Bulk = 5–12 hrs (free)
Glacier, rarely needed, cheap long-term archive →
Deep Archive — 12 hrs standard / 48 hrs bulk. Min 180 days. Cheapest storage.
One Zone-IA risk →
Data is LOST if the single AZ is destroyed. Only use for recreatable or secondary backup data.
