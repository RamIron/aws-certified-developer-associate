# 11.9 - S3 Replication

## Key Concepts

### 1. Two Types of Replication

| Type | Full Name | Regions |
|------|-----------|---------|
| **CRR** | Cross-Region Replication | Different regions |
| **SRR** | Same-Region Replication | Same region |

### 2. How It Works

- Replication is **asynchronous** (happens in the background)
- **Versioning must be enabled** on both source and destination buckets
- Can replicate across **different AWS accounts**
- Must give proper **IAM permissions** to the S3 service (read from source, write to destination)

```
Source Bucket (us-east-1) ──async replication──► Destination Bucket (eu-west-1)
  [versioning ON]                                  [versioning ON]
```

### 3. Use Cases

| Type | Use Cases |
|------|-----------|
| **CRR** | Compliance, lower latency access in another region, cross-account replication |
| **SRR** | Log aggregation from multiple buckets, live replication between production and test accounts |

---

### 4. Replication Notes

- Only **new objects** are replicated after enabling (use **S3 Batch Replication** for existing objects)
- **Delete markers** can optionally be replicated (opt-in setting)
- **Permanent deletes** (by version ID) are **NOT replicated** (prevents malicious deletes spreading)
- Replication is **NOT transitive** — Bucket 1 → Bucket 2 → Bucket 3 does NOT mean Bucket 1 → Bucket 3

```
Bucket 1 ──replication──► Bucket 2 ──replication──► Bucket 3
    │                                                    │
    └──────────── ✕ NOT replicated to Bucket 3 ──────────┘
```

---

## Exam Tips
- Versioning **must be enabled** on both source and destination
- CRR = different regions, SRR = same region
- Replication is **asynchronous**
- Works across different AWS accounts
- Only new objects replicated — use **S3 Batch Replication** for existing ones
- Delete markers: optional replication. Permanent deletes (version ID): never replicated
- **Version IDs are replicated** — same ID in source and destination
- Replication is **NOT transitive** (like VPC Peering!)
- CRR signal: "compliance", "lower latency in another region"
- SRR signal: "aggregate logs", "replicate prod to test"
