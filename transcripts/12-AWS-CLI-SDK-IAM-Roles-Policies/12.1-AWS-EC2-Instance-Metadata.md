# 12.1 - AWS EC2 Instance Metadata

## Key Concepts

### 1. What is IMDS?

- **IMDS** = Instance Metadata Service
- Allows EC2 instances to **learn about themselves** without using an IAM Role
- Access via a special URL: `http://169.254.169.254`

### 2. What Can You Access?

| Can Access | Cannot Access |
|-----------|--------------|
| Instance name | IAM **policy** attached to the role |
| Public IP | |
| Private IP | |
| IAM Role **name** | |
| Temporary credentials | |
| User data (launch script) | |

**Key distinction:** You can get the IAM Role **name**, but NOT the IAM **policy** content.

### 3. Metadata vs User Data

| Term | Meaning |
|------|---------|
| **Metadata** | Information about the EC2 instance (name, IPs, role, etc.) |
| **User Data** | The launch script of the EC2 instance |

Both are accessible through the same URL (`169.254.169.254`).

### 4. IMDSv1 vs IMDSv2

| Version | How It Works | Security |
|---------|-------------|----------|
| **IMDSv1** | Direct access to the URL — one step | Less secure |
| **IMDSv2** | Two steps: get token (PUT), then call URL with token as header | More secure (default since 2023) |

```
IMDSv1:
  GET http://169.254.169.254/latest/meta-data/  →  response

IMDSv2:
  Step 1: PUT → get session token
  Step 2: GET with token header → response
```

---

## Exam Tips
- IMDS URL: **169.254.169.254** — memorize this
- Can retrieve IAM Role **name** but NOT the IAM **policy**
- Metadata = info about instance | User Data = launch script
- IMDSv2 is **default since 2023** — requires token (PUT first, then GET with header)
- IMDSv2 is more secure than IMDSv1
