Topic 1 — EC2 Instance Metadata (IMDS)
How Instances Learn About Themselves
Core concept: IMDS lets an EC2 instance learn about itself without an IAM Role. The data is available at the link-local address http://169.254.169.254 — only reachable from inside the instance.
What you CAN retrieve
  • Instance ID, type, AMI ID
  • Public IP and private IP
  • IAM Role name
  • Temporary credentials for the role
  • User Data (launch script)
What you CANNOT retrieve
  • IAM Policy attached to the role
  • Any data from other instances

Exam trap: IMDS gives the role name, never the policy document.

Metadata vs User Data: Metadata = information about the instance (IP, role, type). User Data = the bootstrap script that ran at launch. Both accessible via 169.254.169.254 but at different paths.
IMDSv1 vs IMDSv2
Version How It Works Security Status
IMDSv1 Direct GET to /latest/meta-data/ Less secure — no session required Still works but discouraged
IMDSv2 PUT to get token → GET with token header Session-oriented, resistant to SSRF Default since 2023 (recommended)
IMDSv2 Request Flow
Step 1
›
PUT /latest/api/token
X-aws-ec2-metadata-token-ttl-seconds: 21600
›
Token returned
Step 2
›
GET /latest/meta-data/
X-aws-ec2-metadata-token: <token>
›
Metadata returned
Topic 2 — AWS CLI with MFA
Using MFA Authentication from the CLI
Key mechanism: To make CLI calls that require MFA, you must call STS GetSessionToken. This returns temporary credentials you use for subsequent calls. Your long-term credentials alone are not enough.
1
Call aws sts get-session-token with --serial-number <MFA-device-ARN> and --token-code <6-digit-code>
2
STS returns: AccessKeyId + SecretAccessKey + SessionToken + Expiration
3
Set AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN from the response and make your API calls
Exam pattern: "MFA with CLI" always points to STS GetSessionToken. The serial number is the ARN of the virtual MFA device registered with the IAM user.

Topic 5 — AWS Signature Version 4 (SigV4)
How Every AWS API Request Gets Authenticated
What is SigV4? Every request to AWS APIs must be cryptographically signed using SigV4. The AWS CLI and SDKs handle this automatically — you only need to understand it for custom HTTP clients and pre-signed URLs.
Method 1 — Authorization HTTP Header
Authorization: AWS4-HMAC-SHA256
Credential=AKID/20231201/us-east-1/s3/aws4_request,
SignedHeaders=host;x-amz-date,
Signature=abc123...
Used by the AWS CLI and SDK for all regular API calls.
Method 2 — Query String Parameter
https://bucket.s3.amazonaws.com/file.pdf
?X-Amz-Algorithm=AWS4-HMAC-SHA256
&X-Amz-Credential=...&X-Amz-Expires=3600
&X-Amz-Signature=abc123...
Used by S3 pre-signed URLs. The signature is embedded directly in the URL.
Exam shortcut: "Pre-signed URL" → always uses X-Amz-Signature in the query string (Method 2). Regular SDK/CLI calls → always use the Authorization HTTP header (Method 1).
Exam Quick-Fire Tips
Scenario: "What is the IMDS URL?"
→ 169.254.169.254 (link-local, instance-only)
Scenario: "IMDS returns the role — can it return the policy?"
→ NO. Role name only — never the policy document.
Scenario: "IMDSv2 requires…"
→ PUT to get session token first, then GET with X-aws-ec2-metadata-token header
Scenario: "Use MFA with CLI"
→ Call STS GetSessionToken with --serial-number + --token-code
Scenario: "App gets ThrottlingException"
→ Implement Exponential Backoff (doubles wait: 1s → 2s → 4s...)
Scenario: "When should I retry?"
→ Retry on 5XX errors and ThrottlingException. NEVER retry 4XX (fix the code).
Scenario: "SDK vs custom HTTP — who handles backoff?"
→ SDK has it built-in. Custom HTTP calls = you implement it yourself.
Scenario: "EC2 app has too many IAM permissions"
→ Leftover environment variables (AWS_ACCESS_KEY_ID etc.) override Instance Profile — unset them.
Scenario: "Credentials from env vars vs Instance Profile"
→ Env vars WIN (higher priority). Instance Profile is lowest priority.
Scenario: "App running inside AWS — where to put credentials?"
→ IAM Roles (never store keys in code or env vars inside AWS)
Scenario: "App running outside AWS — where to put credentials?"
→ Environment variables or named profiles (~/.aws/credentials)
Scenario: "Pre-signed URL — what signature mechanism?"
→ SigV4 with X-Amz-Signature in the query string (not in header)
