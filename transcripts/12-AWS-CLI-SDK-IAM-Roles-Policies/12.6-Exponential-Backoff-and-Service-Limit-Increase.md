# 12.6 - Exponential Backoff & Service Limit Increase

## Key Concepts

### 1. Two Types of AWS Limits (Quotas)

| Type | What it limits | Example |
|------|---------------|---------|
| **API Rate Limits** | How many API calls per second | EC2 DescribeInstances: 100 calls/sec, S3 GetObject: 5,500 GET/sec/prefix |
| **Service Quotas** | How many resources you can run | On-Demand Standard Instances: 1,152 vCPUs |

### 2. What to Do When You Hit Limits

| Situation | Solution |
|-----------|---------|
| **Intermittent errors** (occasional throttling) | Use **Exponential Backoff** |
| **Consistent errors** (heavy usage, always over limit) | Request **API throttling limit increase** |
| **Need more resources** | Request **Service Quota increase** (ticket or Service Quotas API) |

### 3. Exponential Backoff

- Strategy: each retry **doubles** the wait time
- 1s → 2s → 4s → 8s → 16s...
- Result: less load on the server over time, allowing it to recover

```
Retry 1: wait 1 second
Retry 2: wait 2 seconds
Retry 3: wait 4 seconds
Retry 4: wait 8 seconds
Retry 5: wait 16 seconds
```

### 4. When to Use Exponential Backoff

- On **ThrottlingException** errors
- On **5XX server errors** (retry makes sense — server issue)
- **NOT on 4XX client errors** (don't retry — your request is wrong)

### 5. SDK vs Custom Implementation

| Approach | Who handles retries? |
|----------|---------------------|
| **AWS SDK** | Exponential Backoff is **built-in** automatically |
| **Custom HTTP calls** | **You** must implement it yourself |

---

## Exam Tips
- ThrottlingException → **Exponential Backoff**
- Retry only on **5XX** errors, never on **4XX**
- AWS SDK already includes Exponential Backoff
- Consistent throttling → request **limit increase**
- Service Quotas can be increased via ticket or **Service Quotas API**
