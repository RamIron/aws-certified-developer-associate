# 12.7 - AWS Credentials Provider & Chain

## Key Concepts

### 1. Credentials Provider Chain (Priority Order)

The CLI/SDK looks for credentials in this order — **first match wins**:

| Priority | CLI | SDK (e.g. Java) |
|----------|-----|-----------------|
| 1 (highest) | Command line options (`--region`, `--access-key-id`) | Java system properties |
| 2 | Environment variables (`AWS_ACCESS_KEY_ID`, etc.) | Environment variables |
| 3 | CLI credentials file — `~/.aws/credentials` (keys & secrets) | Default credential profile file |
| 4 | CLI configuration file — `~/.aws/config` (region, output format) | — |
| 5 | Container credentials (ECS tasks) | Container credentials (ECS) |
| 6 (lowest) | EC2 Instance Profile credentials | Instance Profile credentials |

### 2. The Classic Exam Scenario

```
EC2 instance with:
  ├── Environment variables → IAM user with S3FullAccess (too broad!)
  └── Instance Profile → IAM role with access to ONE S3 bucket (correct)

Result: App still has S3FullAccess — WHY?
Answer: Environment variables have HIGHER priority than Instance Profile
Fix:    UNSET the environment variables → Instance Profile takes over
```

### 3. Credentials Best Practices

- **NEVER** store credentials in your code
- **Inside AWS** → use IAM Roles (EC2 Instance Role, ECS Role, Lambda Role)
- **Outside AWS** → use environment variables or named profiles (`aws configure`)

---

## Exam Tips
- Environment variables have **higher priority** than EC2 Instance Profile
- If an EC2 instance has both env vars and an instance profile, **env vars win**
- "App on EC2 has more permissions than expected" → check for leftover env vars
- **Never store credentials in code** — always inherit from the credentials chain
- Inside AWS = IAM Roles | Outside AWS = env vars or profiles
