# 18.8 - CloudFormation - Outputs & Exports

## Key Concepts

### 1. What Are Outputs?
- The Outputs section is **optional**
- Used to declare values that can be:
  - **Viewed** in the AWS console or via CLI
  - **Exported** and imported by other stacks
- Best use case: network stack exports VPC IDs and Subnet IDs → reused by application stacks

### 2. Output with Export Block
```yaml
Outputs:
  SSHSecurityGroup:
    Description: The SSH Security Group for our company
    Value: !Ref MySSHSecurityGroup
    Export:
      Name: SSHSecurityGroup   # Must be unique per region
```

- `Value` — the actual value being output (usually a `!Ref` or `!GetAtt`)
- `Export.Name` — the name other stacks use to import this value
- Export names must be **unique within a region**

### 3. Importing Outputs — Fn::ImportValue
To use an exported value in another stack:

```yaml
Resources:
  MyEC2Instance:
    Type: AWS::EC2::Instance
    Properties:
      SecurityGroups:
        - !ImportValue SSHSecurityGroup  # References the exported value from another stack
```

- Shorthand: `!ImportValue`
- Full form: `Fn::ImportValue`

### 4. Cross-Stack Dependency & Deletion Rule
- When Stack B imports a value from Stack A, they are **linked**
- **You cannot delete Stack A** until Stack B (and any other stacks referencing it) no longer imports its values
- This prevents accidental deletion of shared infrastructure

```
Stack A (network)          Stack B (application)
  └── Exports: SSHSecurityGroup ──→ ImportValue: SSHSecurityGroup

  ❌ Cannot delete Stack A while Stack B still references it
```

## Exam Tips
- Outputs are **optional** — but essential for cross-stack collaboration
- Export names must be **unique per region**
- Use `!ImportValue` to consume exported values from another stack
- **Cannot delete a stack** that has exported values still being referenced by other stacks
- Best practice: use a dedicated **network stack** to export VPC/Subnet IDs
