# 18.7 - CloudFormation - Mappings

## Key Concepts

### 1. What Are Mappings
- Mappings are **fixed variables** hardcoded inside the CloudFormation template
- Used to differentiate between environments, regions, architectures, etc.
- All possible values are known **ahead of time** and defined in the template
- Safer than parameters when the set of valid values is finite and predictable

### 2. Mapping Structure
```yaml
Mappings:
  RegionMap:
    us-east-1:
      HVM64: ami-0ff8a91507f77f867
      HVMG2: ami-0a584ac55a7631c0c
    us-west-1:
      HVM64: ami-0bdb828fd58c52235
      HVMG2: ami-066ee5fd4a9ef77f1
    eu-west-1:
      HVM64: ami-047bb4163c506cd98
      HVMG2: ami-31c2f645
```

Structure: `MapName → TopLevelKey → SecondLevelKey → Value`

### 3. Accessing Mapping Values — FindInMap
- Use the `!FindInMap` function to retrieve a value
- Syntax: `!FindInMap [ MapName, TopLevelKey, SecondLevelKey ]`

```yaml
Resources:
  MyEC2Instance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [RegionMap, !Ref "AWS::Region", HVM64]
```

The combination of `FindInMap` + `AWS::Region` pseudo parameter makes templates **automatically region-aware**.

### 4. Mappings vs Parameters

| | Mappings | Parameters |
|--|----------|-----------|
| Values defined | In the template (hardcoded) | By the user at deploy time |
| When to use | Values are known in advance and finite | Values depend on user input |
| Control | Safer — you control all valid values | More flexible — user has freedom |
| Examples | AMI IDs per region, env configs | Instance type, DB password |

## Exam Tips
- Mappings = **fixed, hardcoded** values known in advance
- `!FindInMap [ MapName, TopLevelKey, SecondLevelKey ]` is the access function
- Combining `FindInMap` + `AWS::Region` = **automatic region-specific AMI selection**
- Use **mappings** when values are predictable; use **parameters** when values depend on user input
