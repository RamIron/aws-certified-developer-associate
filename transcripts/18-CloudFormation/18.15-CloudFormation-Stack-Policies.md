# 18.15 - CloudFormation - Stack Policies

## Key Concepts

### 1. What Is a Stack Policy?
- A **JSON document** that controls what update actions are allowed on specific resources during a stack update
- Goal: **protect resources against unintentional updates**
- Applied at the stack level, not the resource level

### 2. Default Behavior
- Without a stack policy → **all resources can be updated** (no protection)
- Once a stack policy is set → **all resources are protected by default**
- You must add **explicit Allow statements** for resources you want to be updatable

### 3. Stack Policy Structure

```json
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "LogicalResourceId/ProductionDatabase"
    }
  ]
}
```

- Statement 1: Allow all updates on all resources
- Statement 2: Deny all updates on the `ProductionDatabase` resource
- Result: everything can be updated **except** the production database

### 4. Key Rule — Deny Takes Priority
- When a stack policy is set, the default is **Deny all**
- You need explicit **Allow** for every resource you want to update
- A **Deny** always overrides an **Allow**

## Exam Tips
- Stack policies = **protect resources from unintentional updates** during stack updates
- They are **JSON documents** (not YAML)
- Once set → default is **deny all** → must explicitly allow what can be updated
- Typical pattern: Allow `Update:*` on `*`, then Deny on specific critical resources
- Stack policies only apply to **stack updates** — not stack creation or deletion
