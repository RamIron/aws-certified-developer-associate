# 18.14 - CloudFormation - Deletion Policy

## Key Concepts

### 1. What Is DeletionPolicy?
- A setting applied to **individual resources** in a CloudFormation template
- Controls what happens to a resource when:
  - The resource is **removed from the template**
  - The **entire stack is deleted**
- Applied per-resource, not per-stack

### 2. The Three Deletion Policies

| Policy | Behavior | Default? |
|--------|---------|---------|
| `Delete` | Resource is deleted when stack is deleted | ✅ Yes (default) |
| `Retain` | Resource is preserved — CloudFormation skips deletion | ❌ No |
| `Snapshot` | A final snapshot is taken before the resource is deleted | ❌ No |

### 3. DeletionPolicy: Delete (Default)
```yaml
MyEC2Instance:
  Type: AWS::EC2::Instance
  DeletionPolicy: Delete   # default — explicit but not required
  Properties: ...
```
- Default behavior — no need to specify unless you want to be explicit
- **Exception for S3 buckets:** `Delete` only works if the bucket is **empty**
  - If not empty → deletion fails
  - Fix: manually empty the bucket first, OR use a Custom Resource to empty it automatically

### 4. DeletionPolicy: Retain
```yaml
MyDynamoDBTable:
  Type: AWS::DynamoDB::Table
  DeletionPolicy: Retain
  Properties: ...
```
- Resource stays alive after stack deletion
- CloudFormation logs it as "delete skipped"
- You must **manually delete** the resource if you want it gone
- Works with **any resource type**

### 5. DeletionPolicy: Snapshot
```yaml
MyRDSInstance:
  Type: AWS::RDS::DBInstance
  DeletionPolicy: Snapshot
  Properties: ...
```
- Takes a **final snapshot before deleting** the resource
- The resource itself is deleted; the snapshot remains
- You must **manually delete the snapshot** to fully clean up

**Supported resource types for Snapshot:**
- EBS Volumes
- ElastiCache Cluster / ReplicationGroup
- RDS DBInstance / DB Cluster
- Redshift
- Neptune
- DocumentDB

### 6. Visual Summary
```
Stack deleted
      ↓
Resource with DeletionPolicy: Delete   → gone ❌
Resource with DeletionPolicy: Retain   → stays ✅ (must manually delete)
Resource with DeletionPolicy: Snapshot → snapshot created ✅, resource gone ❌
```

## Exam Tips
- Default DeletionPolicy is **Delete**
- `Delete` on S3 → **fails if bucket is not empty**
- `Retain` → resource survives stack deletion, must be manually cleaned up
- `Snapshot` → final backup taken, resource still gets deleted
- `Snapshot` is only supported on **stateful** resources (EBS, RDS, ElastiCache, Redshift, Neptune, DocumentDB)
