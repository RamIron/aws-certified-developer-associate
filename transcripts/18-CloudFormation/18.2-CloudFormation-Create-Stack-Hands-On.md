## Key Takeaways

### Console upload → S3 automatically
When you upload a template file via the AWS Console, AWS automatically puts it in S3 and CloudFormation references it from there. You don't need to manually upload to S3 — the console does it for you.

### Application Composer
- Visual tool to see your CloudFormation template as a diagram
- Shows resources (EC2, SG, RDS, etc.) and how they connect
- You can toggle between Canvas (visual) and Template (YAML/JSON code)
- Useful for architecture diagrams and understanding existing templates

### Minimal EC2 template structure
```yaml
Resources:
  MyInstance:
    Type: AWS::EC2::Instance
    Properties:
      AvailabilityZone: us-east-1a
      ImageId: ami-xxxxxxxxxxxxxxxxx
      InstanceType: t2.micro
```
- `Resources` is the only mandatory section
- Resource name (`MyInstance`) = logical ID
- `Type` follows the pattern `AWS::<service>::<resource>`

### Auto-tagging
CloudFormation automatically tags every resource it creates with:
- `aws:cloudformation:stack-name` → stack name
- `aws:cloudformation:logical-id` → resource's logical ID in the template
- `aws:cloudformation:stack-id` → full stack ARN

This is how you track costs per stack.

### CloudFormation Console tabs
- **Events** — creation/update/delete history
- **Resources** — all resources in the stack with links to their AWS console pages
- **Outputs** — exported values (empty if none defined)
- **Parameters** — input values used
- **Template** — the exact template as uploaded

## Exam Tips
- Console upload → file goes to S3 automatically
- Application Composer = visual designer for CloudFormation templates
- Resources are auto-tagged with stack name, logical ID, stack ID
- Logical ID = name you give a resource inside the template (e.g., `MyInstance`)
