# 18.11 - CloudFormation - Rollbacks

## Key Concepts

### 1. Stack Creation Failure
When a stack creation fails, you have two options:

| Option | Behavior |
|--------|---------|
| **Roll back all stack resources** (default) | Everything gets deleted. Logs are available to investigate. Resources are gone. |
| **Preserve successfully provisioned resources** | Successfully created resources are kept. Failed resources are rolled back. Useful for troubleshooting. |

> ⚠️ If you preserve resources, you **cannot update the stack to fix it** — you must delete the stack entirely to clean up.

### 2. Stack Update Failure
When a stack update fails:
- CloudFormation **automatically rolls back** to the last known working state (default)
- Anything newly created during the failed update gets deleted
- Logs and error messages are available for investigation

### 3. Rollback Failure — ContinueUpdateRollback
If a rollback itself fails (usually because resources were **manually changed** outside CloudFormation):

1. Fix the affected resources manually
2. Issue a `ContinueUpdateRollback` to retry the rollback

```
Stack update fails
      ↓
Rollback starts
      ↓
Rollback fails (manual resource changes detected)
      ↓
Fix resources manually
      ↓
ContinueUpdateRollback (console / CLI / API)
```

**ContinueUpdateRollback** can be triggered via:
- AWS Console
- AWS CLI
- CloudFormation API

## Exam Tips
- **Creation failure default** → everything rolls back and gets deleted
- **Preserve option** → keeps successfully provisioned resources for troubleshooting, but stack must be deleted afterward
- **Update failure default** → auto rollback to last known working state
- **Rollback failure** → fix manually + call `ContinueUpdateRollback`
- Root cause of rollback failure → resources were **manually modified** outside CloudFormation
