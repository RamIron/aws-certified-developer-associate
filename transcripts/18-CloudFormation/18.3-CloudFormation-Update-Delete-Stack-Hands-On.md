# 18.3 - CloudFormation Update & Delete Stack - Hands On

## Key Takeaways

### Updating a stack
- Must **replace the current template** — you cannot edit the existing one in place
- Upload a new template version → CloudFormation compares old vs new

### Change Sets
- Before applying an update, CloudFormation shows a **Change Set Preview**
- Lists every resource that will be: Added / Modified / Removed / Replaced
- Two types of changes:
  - **Replacement: false** → resource updated in place (no interruption)
  - **Replacement: true** → old resource terminated, new one created to replace it

### Parameters in action
- Parameters defined in the template prompt the user for input at deploy/update time
- The value entered gets injected into the template at runtime
- Example: `SecurityGroupDescription` parameter → becomes the description of the SG

### Delete order
- CloudFormation determines the **correct order to delete resources automatically**
- Example: Elastic IP deleted first → EC2 terminated next → Security Groups last
- Same intelligence applies to creation order

### Why delete via CloudFormation (not manually)
Manually terminating resources one by one leaves orphans (SGs, EIPs still exist).
Deleting the **stack** removes everything in the correct order — clean teardown.

## Exam Tips
- Updating a stack → must upload a **new template version**
- **Change Set** = preview of what will change before applying an update
- **Replacement: true** → resource is terminated and recreated (not updated in place)
- Always delete via the **stack** (not individual resources) for clean cleanup
- CloudFormation handles creation AND deletion order automatically
